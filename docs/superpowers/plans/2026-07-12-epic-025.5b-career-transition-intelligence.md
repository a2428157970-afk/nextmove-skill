# Epic-025.5B Career Transition Intelligence Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build an internal, evidence-grounded Career Transition Intelligence model that classifies career moves, calibrates risk, and recommends gap-linked actions without changing any public API.

**Architecture:** Add immutable transition contracts and a stateless orchestrator under `skill/career/`. Keep transition-type/gap aggregation in `transition.py`, risk calibration in `transition_risk.py`, and the bounded action catalog in `transition_actions.py`. The benchmark composes existing classifier, stage, transfer, and explanation outputs with the new assessor; `JobMatcher` and its six-field public result remain unchanged.

**Tech Stack:** Python standard library, frozen/slotted dataclasses, enums, existing `unittest`, existing JSON benchmark fixtures; no dependency, provider, database, or network change.

## Global Constraints

- Do not modify `JobMatchResult`, `NextMoveSkill.run()`, public Skill schemas, `skill/agent/`, Agent Runtime, Application Layer, CLI, providers, dependency manifests, databases, network behavior, or Git tags.
- Preserve exactly these public fields and serialized order: `match_score`, `matched_skills`, `missing_skills`, `strengths`, `gaps`, `recommendations`.
- Keep `CareerTransitionAssessment` internal; do not export it from the public `skill` package or Agent schema.
- Consume `CareerStageAssessment`, `TransferableSkillAssessment`, and `MatchExplanationResult` without mutating them.
- Preserve `matched`, `partial`, `missing`, and `unknown`; never convert unknown evidence into an asserted missing capability.
- Store only canonical names, bounded reasons, future actions, and filtered `EvidenceItem` values. Never retain resume text, JD text, identity, employer, credential, endpoint, or provider content.
- Risk describes evidence conditions only. Never output “不适合”, “无法转型”, “unsuitable”, “cannot transition”, or equivalent fatalistic conclusions.
- Every recommended action must reference one structured gap and state the direct evidence it is intended to create.
- Keep adjacent mappings directional and non-transitive. Administration-to-HR is adjacent; Sales-to-Product remains cross-domain.
- Use TDD for every production behavior: failing test, verified red, minimal implementation, verified green, then focused commit.

- [ ] **Step 4: Implement ordered risk rules**

~~~python
class TransitionRiskAssessor:
    def assess(
        self,
        transition_type: TransitionType,
        direct_evidence: tuple[EvidenceItem, ...],
        transferable_evidence: tuple[EvidenceItem, ...],
        gaps: tuple[TransitionCapabilityGap, ...],
    ) -> TransitionRiskAssessment:
        core_gaps = tuple(gap for gap in gaps if gap.core)
        if transition_type == TransitionType.UNKNOWN:
            return self._unknown()
        if not direct_evidence and not transferable_evidence and len(core_gaps) >= 2:
            return self._high("No grounded bridge evidence is available.", core_gaps)
        if (
            transition_type == TransitionType.CROSS_DOMAIN
            and len(direct_evidence) < 2
            and len(core_gaps) >= 2
        ):
            return self._high("The cross-domain path is transfer-led with sparse direct evidence.", core_gaps)
        if (
            transition_type in {TransitionType.SAME_ROLE, TransitionType.SAME_DOMAIN}
            and not core_gaps
            and len(direct_evidence) >= 2
        ):
            return self._low()
        return self._medium(transition_type, direct_evidence, transferable_evidence, core_gaps)
~~~

Helper methods must construct bounded factors and list canonical core-gap names.
Do not calculate career-success probability.
