# Epic-020.1 Application Layer Boundary Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a standalone, fail-fast Product Application Layer that orchestrates the stable public Skill API into one career-analysis response.

**Architecture:** `CareerAnalysisService` receives application requests only and delegates them to `CareerAnalysisWorkflow`. The workflow calls `NextMoveSkill.run()` in the fixed four-step sequence, stops at the first unsuccessful `SkillResponse`, and aggregates successful results into application-owned schemas. Dependencies flow from application to public Skill contracts only.

**Tech Stack:** Python 3.11+, standard-library `dataclasses`, `unittest`, and the existing `skill.utils.to_dict` serializer.

## Global Constraints

- Do not modify files under `skill/`, `backend/`, or `frontend/`.
- Do not add dependencies, FastAPI, web UI, database, authentication, user sessions, AI providers, MCP, or Agent Adapter changes.
- Call every capability exclusively through `NextMoveSkill.run(capability, payload)`.
- Do not expose resume input, job-description input, prompts, credentials, provider metadata, or provider responses in application output.
- Stop on the first unsuccessful Skill response and retain its original structured `SkillError`.
- Verify with `python -m unittest discover -s tests -v`, `python -m compileall application skill tests`, and `git diff --check`.
