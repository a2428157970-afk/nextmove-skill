# Epic-025.5A Career Taxonomy Expansion Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Expand NextMove's internal occupational knowledge with a Product domain, Product job families and capabilities, plus bounded Sales-to-Product and Administration-to-HR transferable-skill mappings, while preserving every public contract.

**Architecture:** Keep domain, family, and Product capability vocabulary in `taxonomy.py`; add an internal transfer assessment schema and a stateless, directional assessor; let `MatchScorer` combine direct requirement evidence with transfer evidence without changing the four evidence statuses; let `MatchExplanationBuilder` consume the internal assessment without exposing it through `JobMatchResult`. Update the Epic-025.4 synthetic benchmark to verify the two transition paths and their safety constraints.

**Tech Stack:** Python standard library, frozen/slotted dataclasses, enums, existing `unittest` suite, existing JSON benchmark fixtures; no dependency, provider, database, network, or public API change.

## Global constraints

- Do not modify `NextMoveSkill.run()`, `JobMatchResult`, `skill/agent/`, the Agent Contract, Application Layer, CLI, providers, dependency manifests, or Git tags.
- Preserve the six `JobMatchResult` fields and their serialized order: `match_score`, `matched_skills`, `missing_skills`, `strengths`, `gaps`, `recommendations`.
- Preserve the evidence states `matched`, `partial`, `missing`, and `unknown`.
- A transferable capability may support only `partial`; it must never produce `matched`, direct Product experience, or a `matched_skills` entry.
- `unknown` must never be rewritten as `missing` merely because transfer evidence is absent.
- Direct target-role evidence takes precedence over transferable evidence for the same requirement.
- Sales customer management is not Product management. Administration is not HR expertise.
- Keep mapping rules directional, family/capability scoped, non-transitive, deterministic, offline, and provider-neutral.
- Store only filtered evidence snippets already accepted by the evidence layer. Do not store resume text, JD text, credentials, endpoints, or personal information.
- Use TDD for every behavior change: focused failing test, minimal implementation, focused green test, then commit.

## Planned file map

- Modify `skill/matching/taxonomy.py`: Product domain, Product job families, Product capability categories and bounded signals.
- Modify `skill/matching/schemas.py`: internal transferable-capability and assessment contracts; optional internal assessment on `MatchAssessment`.

- Modify `tests/test_matching_classifier.py`: Product classification and false-positive coverage.

- Create `tests/test_matching_transferability.py`: internal model, Sales-to-Product, and Administration-to-HR rules.
- Modify `tests/test_matching_explanation_builder.py`: transfer strengths, gaps, risks, privacy, and empty evidence.
- Modify `tests/test_matching_facade_explanation.py`: facade integration and exact six-field contract.
- Modify `tests/benchmark/scenarios/03-sales-to-product-manager.json`: Product-transition expectations.
- Modify `tests/benchmark/scenarios/04-administrative-assistant-to-hr-assistant.json`: adjacent-transition expectations.

- Modify project records only after implementation verification: `CHANGELOG.md`, `PROJECT.md`, and `NEXT_TASK.md`.
