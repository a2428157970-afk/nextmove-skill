# Epic-026.1 Skill Distribution Package Design

## 1. Purpose and scope

Epic-026.1 designs a self-contained distribution package that lets a non-Python
user obtain NextMove, understand whether their Agent can load it, and complete a
first evidence-grounded career analysis in less than five minutes.

This Epic changes distribution and usage experience only. It does not change:

- Skill Core, `NextMoveSkill.run()`, or any matching, career, transition, or
  reporting behavior;
- `skill.json`, the Agent Contract, or existing public schemas;
- the Application Layer, CLI behavior, providers, backend, or frontend;
- dependency manifests, databases, network services, or user accounts.

The implementation Epic will assemble documentation and fictional sample
assets from existing, frozen runtime artifacts. It will not create a second
runtime, an Agent adapter, or a platform-specific integration.

## 2. Adoption Gap Analysis

### 2.1 Current acquisition path

The current repository contains the right ingredients, but they are optimized
for maintainers rather than first-time users:

| Current asset | What it provides | First-use gap |
|---|---|---|
| `SKILL.md` | Capability, input, output, and limitation contract | Assumes the reader already has the repository and knows how an Agent discovers a Skill. |
| `skill.json` | Machine-readable Skill metadata and schemas | Useful to runtimes and developers, but not an installation guide. |
| `README.md` | Project overview, installation, CLI, Python, and five-minute trial | The primary path starts from a cloned repository and Python installation; “Agent has access” is not explained. |
| `docs/quick-start.md` | A concise pilot flow | Tells the user to use an Agent that can access NextMove without showing platform-specific loading or a loading check. |
| `docs/agent-prompt-template.md` | Safe prompts for current capabilities | Prompts are repository-relative and do not explicitly request the Human Career Report or a career-transition report. |
| `docs/agent-compatibility.md` | Honest contract/repository/runtime status | Separate from the quick start, so a new user can mistake contract compatibility for verified execution. |
| `examples/skill_demo.py` | Direct Python invocation | Requires Python knowledge and is unsuitable as the default adoption path. |
| `examples/data/*` | Fictional technology sample | Demonstrates one technical profile, not HR, technology, and cross-domain transition coverage. |
| `docs/pilot-feedback-template.md` | Privacy-aware pilot feedback | Not discoverable from a standalone package. |

### 2.2 User-visible gaps

1. **Download entry:** there is no named, versioned artifact that a user can
   download without cloning the repository.
2. **File organization:** installation instructions, prompts, samples, license,
   and runtime contract are spread across repository paths.
3. **Agent loading:** “uploading files,” “installing a Skill,” and “executing
   NextMove” are not distinguished.
4. **Prompt selection:** the user must edit repository examples and infer which
   prompt fits full analysis, matching, or transition.
5. **First success:** there is no required preflight response proving that the
   Agent can invoke NextMove rather than imitate it.
6. **Examples:** the current data demonstrates only a technology profile and is
   not organized as a nontechnical walkthrough.
7. **Trust:** version pairing, verification date, platform status, and safe
   feedback rules are not visible in one place.

### 2.3 Adoption requirements

The package must therefore be:

- downloadable as one versioned archive;
- usable without reading Python, cloning Git, or running the CLI;
- self-contained after extraction, with only package-local links;
- explicit about whether a platform can read files, discover a Skill, and
  actually execute NextMove;
- safe by default: no fabricated experience, no hidden inference, no employment
  guarantee, and no request for unnecessary personal data;
- testable through a fictional first run before a user uploads personal data.

## 3. Selected package design

The release artifact is a documentation-first wrapper around the unchanged
NextMove Skill. It is not another Python distribution.

```text
nextmove-skill-package/
├── SKILL.md
├── skill.json
├── PACKAGE_MANIFEST.json
├── QUICK_START.md
├── README.md
├── PROMPTS.md
├── examples/
│   ├── sample_resume.txt
│   └── sample_job_description.txt
└── LICENSE
```

The downloadable filename is:

```text
nextmove-skill-package-<package_version>.zip
```

### 3.1 File responsibilities

| File | Responsibility | Source policy |
|---|---|---|
| `SKILL.md` | Agent-readable NextMove capability and invocation contract | Byte-for-byte copy of the release's canonical root file. |
| `skill.json` | Existing Skill metadata and public schemas | Byte-for-byte copy; no distribution-only fields are added. |
| `PACKAGE_MANIFEST.json` | Package format, package version, included Skill version, and file inventory | Distribution-owned metadata; never imported by Skill Core. |
| `QUICK_START.md` | One path from download to a fictional Human Career Report | Written for nontechnical users; package-local references only. |
| `README.md` | Package purpose, contents, platform status, loading guide, troubleshooting, feedback, and privacy | Distribution entry point after extraction. |
| `PROMPTS.md` | Three copy-ready prompts | Full analysis, job matching, and career transition. |
| `examples/sample_resume.txt` | Fully fictional resume used by the first run | No contact details or real organization names. |
| `examples/sample_job_description.txt` | Fully fictional target role paired with the resume | No copied vacancy or real organization details. |
| `LICENSE` | License terms | Byte-for-byte copy of the canonical repository license. |

`PACKAGE_MANIFEST.json` uses this minimal shape:

```json
{
  "package_name": "nextmove-skill-package",
  "package_format_version": 1,
  "package_version": "0.8.0",
  "skill_name": "nextmove-career-intelligence",
  "skill_version": "0.8.0",
  "files": [
    "SKILL.md",
    "skill.json",
    "QUICK_START.md",
    "README.md",
    "PROMPTS.md",
    "examples/sample_resume.txt",
    "examples/sample_job_description.txt",
    "LICENSE"
  ]
}
```

The manifest intentionally excludes its own path from `files`; the array is the
payload inventory checked after reading the manifest. A validator must reject a
missing, extra, duplicated, absolute, or parent-traversing path.

### 3.2 Single-source and assembly rules

- Runtime artifacts are copied, never edited inside the package build.
- Distribution documents have canonical repository sources under a dedicated
  distribution source directory selected during implementation.
- The generated archive and expanded `nextmove-skill-package/` directory are
  release artifacts, not duplicate hand-maintained source trees.
- Every Markdown link in the package resolves inside the package, except the
  explicitly labelled official platform documentation and feedback URL.
- Archive extraction must produce exactly one top-level directory.
- The archive contains no `.git`, caches, bytecode, test output, local paths,
  personal resume, API key, or environment file.

## 4. Agent Loading Guide

### 4.1 Verification vocabulary

Every platform row and instruction carries one of three evidence levels:

1. **Contract supported:** the platform can consume the documented file or
   Skill format in principle. This says nothing about NextMove execution.
2. **Repository tested:** the package, prompts, samples, and unchanged local
   NextMove entrypoint pass repository validation. This is not a test inside an
   external product.
3. **External runtime verified:** a named product and version loaded the exact
   released package and passed the dated adoption test, including a negative
   “Skill unavailable” case. Only recorded evidence may grant this label.

Status is cumulative but never inferred. “Contract supported” must not be
rendered as “works,” and a file upload must not be rendered as a Skill install.

### 4.2 Platform matrix at design baseline

| Platform | Documented loading surface | Baseline status | User-facing guidance |
|---|---|---|---|
| ChatGPT Agent | ChatGPT can read uploaded files and Project/GPT knowledge, but file access alone does not install or execute NextMove | Contract supported for reference files; external runtime verification pending | Uploading the package may provide instructions and examples only. Proceed with a career analysis only if a configured runtime explicitly confirms that `nextmove-career-intelligence` is available; otherwise stop and report that the Skill cannot be invoked. |
| Claude Code | Personal or project Skills discovered from a directory containing `SKILL.md` | Contract supported; external runtime verification pending | Copy the complete package to the documented Skill directory for the user's installed Claude Code version. Preserve all sibling files. Ask Claude to identify the Skill and run the availability preflight before sending resume data. |
| Cursor | Agent Skills based on `SKILL.md` are supported by current editor/CLI releases | Contract supported; external runtime verification pending | Import or copy the complete folder using the current Cursor Agent Skills instructions. Do not rely on a remote URL or a lone `SKILL.md`; run the availability preflight. |
| Codex | Codex Skills use a folder with `SKILL.md` plus optional bundled resources | Contract supported and repository tested; external package-runtime verification pending | Copy the complete folder to a documented personal or workspace Skill location, start or refresh the task as required by the installed Codex version, then run the preflight. |

The implementation guide links to official, version-current instructions rather
than hard-coding UI labels that may change. As of the design review on
2026-07-13, the authoritative references are:

- OpenAI, [Build skills](https://developers.openai.com/codex/skills);
- OpenAI, [File Uploads FAQ](https://help.openai.com/en/articles/8555545-custom-gpts);
- Anthropic, [Extend Claude with skills](https://code.claude.com/docs/en/skills);
- Cursor, [Agent Skills](https://cursor.com/docs/skills).

### 4.3 Availability preflight

Before any personal resume is supplied, the user sends:

```text
请确认当前会话是否能够实际调用 nextmove-career-intelligence Skill。
只回答以下之一：
1. NEXTMOVE_READY：并列出可调用的 NextMove 能力名称；
2. NEXTMOVE_UNAVAILABLE：说明当前只能读取说明文件，不能执行 Skill。
不要根据 SKILL.md 模拟分析结果。
```

Only `NEXTMOVE_READY` permits the quick start to continue. A vague claim such as
“I have read the files” is not success. `NEXTMOVE_UNAVAILABLE` routes the user
to the platform's verified installation help; it never falls back to a
simulated report.

## 5. One-minute Quick Start design

“One-minute” means one short path that takes less than five minutes under the
following measured conditions: the user already has a supported Agent runtime,
the package is downloaded, and no product sign-up or Python setup is required.

### 5.1 Happy path

| Time budget | User action | Package response/checkpoint |
|---:|---|---|
| 0:00-0:30 | Download and extract the versioned archive | One `nextmove-skill-package/` folder is visible. |
| 0:30-1:30 | Follow the platform row in `README.md` and add the complete folder | User does not select individual runtime files. |
| 1:30-2:00 | Send the availability preflight | Agent returns `NEXTMOVE_READY`; otherwise the flow stops safely. |
| 2:00-2:30 | Copy “完整职业分析” from `PROMPTS.md` | Prompt names NextMove and Human Career Report. |
| 2:30-3:00 | Upload the two fictional sample files | No personal data is used for the first run. |
| 3:00-5:00 | Run and inspect the report | Report separates evidence, gaps, transition, actions, risk, and confidence; insufficient evidence is explicit. |

After the fictional run succeeds, the user may replace the sample resume with a
redacted personal resume. The quick start recommends removing name, phone,
email, address, identifiers, and confidential employer information first.

### 5.2 Success criteria

The first run is successful only when:

- the Agent confirms the real Skill is available;
- the sample facts are not expanded or embellished;
- the result is a Human Career Report, not raw `skill.json` or Python output;
- factual observations and recommendations are visibly separated;
- evidence gaps and missing information remain visible;
- no guarantee or suitability verdict appears.

## 6. Prompt templates

All templates share a fixed safety preamble:

```text
请使用 nextmove-career-intelligence Skill 完成以下任务。
开始前先确认 Skill 可实际调用；如果只能读取说明文件，请明确回答
NEXTMOVE_UNAVAILABLE，不要模拟 Skill 结果。
只使用我提供的材料，不得编造职位、职责、技能、年限、成果或教育经历。
将“材料中的事实”“基于证据的判断”“建议”明确区分。
信息不足时主动列出缺失信息，并降低结论强度。
最终输出 Human Career Report；不要把录用概率或匹配分数写成保证。
```

### 6.1 完整职业分析

```text
[安全前言]

请基于我提供的简历和目标岗位完成完整职业分析。
报告依次包含：你的职业画像、你的优势、岗位匹配、如果转型、下一步行动。
每项优势必须附简历证据；区分直接经验与可迁移能力。
行动按“立即行动 → 短期提升 → 长期发展”排序，并说明行动对应的差距。
最后列出注意事项、置信度和仍需补充的信息。

输入：
- 简历：见上传文件
- 目标岗位：见上传文件；如未提供，请明确说明岗位匹配和转型判断受限
```

### 6.2 岗位匹配

```text
[安全前言]

请比较我提供的简历与目标岗位。
保留 NextMove 的原始匹配结果，不重新估算或美化分数。
说明匹配程度、为什么匹配、主要缺口，并为每个匹配点给出证据。
把“材料未提及”与“已确认不具备”分开，不要把未知信息写成能力缺失。
给出与主要缺口直接关联的下一步行动。

输入：
- 简历：见上传文件
- 目标岗位说明：见上传文件
```

### 6.3 职业转型

```text
[安全前言]

请分析从当前职业到目标职业的转型路径。
明确展示“当前领域 → 目标领域”、转型类型、直接能力、可迁移能力、
缺失能力和需要验证的信息。不得把可迁移能力表述为已经拥有目标岗位
的直接经验，也不得断言我一定成功、一定失败、适合或不适合。
给出与差距关联的“立即行动 → 短期提升 → 长期发展”计划。

输入：
- 简历：见上传文件
- 目标岗位或方向：见上传文件或下方文字
```

In `PROMPTS.md`, `[安全前言]` is expanded in full for every copy-ready prompt;
the user never has to combine fragments manually.

## 7. Sample content strategy

### 7.1 Privacy and fiction rules

Both sample files are wholly fictional and must contain a prominent marker:

```text
FICTIONAL SAMPLE — NOT A REAL PERSON OR JOB POSTING
```

They contain no real name, phone, email, address, social profile, employer,
school, client, vacancy URL, proprietary metric, or copied job-posting text.
Names use role labels such as `Candidate A`, and organizations use neutral
labels such as `Fictional Services Company`.

### 7.2 Coverage within two files

The two required filenames contain three clearly separated mini-scenarios so
coverage does not require undocumented extra files:

| Scenario | Resume section | Job-description section | Learning purpose |
|---|---|---|---|
| HR | `Scenario A — HR Specialist` | `Scenario A — HR Specialist` | Strong same-domain match and evidence-backed strengths. |
| Technology | `Scenario B — Technology Professional` | `Scenario B — Technology Role` | Direct technical capability and an explicit evidence gap. |
| Career Transition | `Scenario C — Sales Professional` | `Scenario C — Product Manager` | Cross-domain path, transferable skills, and missing target capabilities. |

The Quick Start selects Scenario C by name so the Agent cannot accidentally mix
sections. README invites the user to repeat with A and B. Each pair includes
enough detail for an informative report but intentionally leaves one unknown so
the output must demonstrate honest insufficiency handling.

### 7.3 Sample assertions

- Every resume claim appears only in the matching scenario.
- Every job requirement is fictional and maps to at least one direct,
  transferable, missing, or unknown evidence case.
- No scenario contains an employment outcome or a model answer.
- Expected observations live in validation tests, not inside the sample, so the
  Agent is not prompted to copy a predetermined conclusion.

## 8. Version strategy

Package and Skill versions are separate Semantic Versioning streams.

| Version | Meaning | Change examples |
|---|---|---|
| `package_version` | Distribution contents and adoption experience | Quick-start correction, new prompt wording, sample improvement, package layout change. |
| `skill_version` | Included NextMove runtime contract and capability release | Skill Core or public contract release, managed outside this Epic. |
| `package_format_version` | Machine-readable manifest/layout generation | Breaking validator or manifest-shape change. |

Initial pairing is:

```text
Package 0.8.0
contains Skill 0.8.0
package format 1
```

Rules:

1. A documentation-only package patch may release `0.8.1` while continuing to
   contain Skill `0.8.0`; this does not imply a Core API change.
2. A new Skill release requires a newly assembled and validated package, but
   package and Skill numbers need not remain equal after the first release.
3. `skill_version` in the package manifest must equal the canonical version in
   `skill.json` and the runtime's authoritative `skill.__version__` value.
4. The archive filename uses `package_version`; README displays both versions.
5. No package document may claim forward compatibility with an untested Skill
   version.
6. A breaking package layout/installation change increments package major or
   `package_format_version` as appropriate without changing the Core API.

## 9. Feedback and privacy strategy

README offers three issue categories:

- **Bug:** package file missing, link broken, loading step fails, Skill cannot be
  discovered, or output violates a documented invariant.
- **Experience:** download, extraction, wording, platform instructions, prompt
  selection, or time-to-first-report is confusing.
- **Quality:** evidence linkage, gap clarity, transition usefulness, action
  relevance, language safety, or insufficient-information behavior is weak.

The feedback template requests only:

- package version and included Skill version;
- platform, product version, and verification level;
- selected fictional scenario or a redacted issue summary;
- expected versus observed behavior;
- optional sanitized excerpt containing no identifying information.

Before any free-text field, it displays:

```text
请勿提交完整简历、姓名、电话、邮箱、地址、证件号、真实公司或客户机密、
未公开岗位内容、API Key、访问令牌或其他可识别个人/组织的信息。
请使用包内虚构样例复现；如必须描述个人场景，请先脱敏并仅保留必要片段。
```

The package may link to a repository issue template selected during
implementation. It does not collect feedback itself, create accounts, transmit
files, add telemetry, or introduce a network service.

## 10. Validation and adoption test

### 10.1 Static package validation

The implementation must verify:

- exact top-level structure and manifest inventory;
- canonical copies of `SKILL.md`, `skill.json`, and `LICENSE` are unchanged;
- manifest versions are valid and version pairing is consistent;
- archive paths are relative, unique, and traversal-safe;
- all package-local links resolve and official links are labelled external;
- all three full prompts contain the Skill name, no-fabrication rule,
  fact/advice separation, Human Career Report request, and insufficiency rule;
- samples are marked fictional and contain no email, phone, URL, API key, or
  repository-known real personal data;
- protected runtime and application paths have no diff.

### 10.2 Repository adoption test

A clean temporary extraction must be sufficient to follow QUICK_START without
opening the repository. The test records:

1. archive extraction result;
2. first file opened;
3. selected platform path;
4. preflight result;
5. prompt copied without edits;
6. correct sample scenario selected;
7. report completeness and safety checks;
8. elapsed active user time and every point of confusion.

Repository validation can pass packaging and content checks, but it cannot set
an external platform to “External runtime verified.”

### 10.3 External runtime verification gate

For each platform/version, a human tester must use the exact release archive
and record:

- install/load steps and screenshots or sanitized logs;
- `NEXTMOVE_READY` with actual capability discovery;
- successful fictional full-analysis, matching, and transition prompts;
- an unavailable/misconfigured test that returns `NEXTMOVE_UNAVAILABLE` and
  does not simulate a result;
- Human Career Report structure, evidence, insufficiency, and language checks;
- date, product version, package version, and Skill version.

Only after this evidence is reviewed may the README status change to “External
runtime verified.”

## 11. Decisions and rejected alternatives

### 11.1 Selected: one portable archive with honest platform lanes

One package gives users a stable download and keeps the runtime contract,
prompts, samples, and trust information together. Platform-specific guidance
is documentation, not duplicated platform forks.

### 11.2 Rejected: publish only `SKILL.md`

A lone Skill file does not explain version pairing, loading, first prompts,
samples, privacy, or the difference between reference reading and execution.

### 11.3 Rejected: claim universal upload support

Uploading a document can provide context without installing executable Skill
capabilities. A universal claim would encourage fabricated reports and violate
the verified-capability requirement.

### 11.4 Rejected: ship Python setup as the default path

Python remains available to developers through the repository, but it does not
solve the requested nontechnical distribution experience.

### 11.5 Rejected: one platform-specific package per Agent

That would create divergent contracts, duplicated content, and misleading
compatibility promises before external runtime validation exists.

## 12. Non-goals

- publishing the archive or creating a download website in this Epic;
- implementing adapters for ChatGPT, Claude, Cursor, or Codex;
- changing any Skill input, output, schema, or behavior;
- executing personal resumes in test automation;
- adding telemetry, analytics, authentication, storage, or feedback services;
- guaranteeing installation on an untested product version;
- updating release/project closeout files in this design-only commit.
