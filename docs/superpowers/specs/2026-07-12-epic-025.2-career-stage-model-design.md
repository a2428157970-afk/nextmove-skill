# Epic-025.2 Career Stage Model Design

## 1. Decision and scope

Epic-025.2 freezes a deterministic Career Stage Model for Skill Core. It replaces the internal use of experience-entry count as the primary career-progress proxy with an evidence-based assessment of experience, responsibility, and impact.

This Epic is design-only. It does not change `NextMoveSkill.run()`, any Skill API signature, `JobMatchResult`, Application Layer, Agent Tool schema, CLI, AI Provider, database, dependency, frontend, backend, release files, or Git tag. The existing application -> public Skill dependency direction remains unchanged.

The selected approach is an internal `CareerStageAssessment`, mapped back to the existing public `career_level` and `CareerAdviceResult.current_level` values. This preserves public serialization and Agent compatibility while giving analysis and advice richer internal evidence.

## 2. Current career-level architecture

### 2.1 Field definition and output boundary

`skill/schemas/analysis.py` declares:

```python
CareerLevel = Literal["intern", "junior", "mid", "senior", "lead", "unknown"]

@dataclass(slots=True)
class ResumeAnalysisResult:
    ...
    career_level: CareerLevel = "unknown"
```

The Agent `RESUME_ANALYSIS_OUTPUT_SCHEMA` repeats exactly this enum. `career_level` is therefore a public serialized contract, not merely an implementation detail. `CareerAdviceResult.current_level` is a public string field and currently carries the same value.

### 2.2 Current generation and consumption

- `skill/analysis/analyzer.py::ResumeAnalyzer._career_level()` returns `unknown` for no experience, `junior` for one experience entry, `mid` for two, and `senior` for three or more. It does not emit `intern` or `lead`.
- `skill/career/advisor.py::CareerAdvisor._current_level()` returns `analysis.career_level` when known; otherwise it repeats the same experience-count heuristic.
- `CareerAdvisor` selects possible paths from English technical/data/management keyword sets, but stage does not otherwise affect paths, skill gaps, or actions.
- `skill/matching/` does not consume career level. Epic-025.1 internal matching assessment and public `JobMatchResult` must remain independent of this Epic.
- Tests lock empty profiles to `unknown`, three entries to `senior`, a one-entry technical profile to `junior`, and a two-entry manager profile to `mid`. Interface, API, Agent, workflow, serialization, Pilot, and Application tests transport these results.

### 2.3 Risks

Changing the public literal to `entry/developing/experienced/advanced` would break Agent schema validation and consumers interpreting the legacy values. Adding a public `career_stage`, confidence, or signal breakdown field would change serialization and Agent output without a versioned contract decision. Changing stage inference can change legacy `career_level` behavior and advice wording, so legacy fixtures need explicit compatibility assertions.

## 3. Alternatives considered

### A. Replace public `career_level` with the new taxonomy

This is clearer to new consumers but breaks the frozen Agent enum and downstream serialized output. Rejected.

### B. Add optional public stage fields

Optional Python fields still appear in dataclass serialization and diverge from the Agent output schema. Rejected until a separate versioned Agent Contract Epic explicitly approves it.

### C. Internal Career Stage assessment with legacy mapping (selected)

Classify stage and confidence in `skill/career/`, consume it internally in analysis and advice, then map to the existing public strings. This keeps the provider-neutral/offline design and lets tests validate the richer model without widening the public contract.

## 4. Career Stage Taxonomy v1

`CareerStage` is an internal enum with `entry`, `developing`, `experienced`, `advanced`, and `unknown`. Ranges are contextual guides, not gates. A candidate never receives a higher stage merely because of elapsed time, or a lower stage merely because a date is absent.

| Stage | Usual experience range | Responsibility characteristics | Capability signals | Typical roles |
|---|---:|---|---|---|
| `entry` | 0-2 years | Learning, supported delivery, defined tasks | internships, portfolio/projects, foundational tools, supervised execution | intern, junior analyst, assistant, coordinator, junior engineer |
| `developing` | 2-5 years | Independently owns a bounded module, client/account/process segment, or recurring deliverable | ownership language, delivery across a complete cycle, stakeholder coordination, repeatable execution | specialist, analyst, recruiter, accountant, operations specialist, software engineer |
| `experienced` | 4-8 years | Leads complex work or a function, improves a process, influences cross-functional partners | system/process ownership, mentoring, cross-team collaboration, measurable delivery or improvement | senior specialist, senior engineer, finance manager, HRBP, operations manager |
| `advanced` | 7+ years | Sets direction, manages people or broad scope, owns strategic/organisational outcomes | people leadership, portfolio/program responsibility, organisation-wide process design, material business impact | principal, director, head of function, engineering manager, senior manager |
| `unknown` | not determinable | Evidence insufficient or contradictory | no reliable work/project/responsibility evidence | incomplete profile |

Overlaps are deliberate. An early-career candidate with clear people leadership may be `developing`; an individual contributor with extensive expert scope may be `advanced`; a career changer retains demonstrated transferable stage rather than being reset to `entry` solely because the target domain is new.

## 5. Signal Model

### 5.1 Evidence sources

Only resume-provided evidence is used: summary, experience role/title/date/highlights, projects, skills, education, certifications, languages, and raw text. Personal contact data is excluded. The model must never infer age, seniority, management scope, salary, employer size, or results that are not stated.

### 5.2 Experience signals

- **Duration:** parse explicit experience dates or explicit stated years. Date duration is evidence, not a hard threshold; unknown/overlapping dates receive no invented total.
- **Project complexity:** explicit multi-system, full-cycle, program, architecture, audit, vendor, client, or process language increases evidence only when associated with a supplied role/project.
- **Management scope:** explicit direct reports, hiring, coaching, mentoring, delivery ownership, budget, portfolio, or team language is a responsibility signal, not title-only proof.

### 5.3 Responsibility signals

- owning a module, account, workflow, or recurring operating process;
- defining or improving a process, standard, control, or system;
- cross-functional or stakeholder coordination;
- leading, mentoring, managing, or developing people.

Signals should be drawn from focused English and Chinese aliases with bounded phrase matching, like Epic-025.1. Exact responsibility phrases are stronger than broad verbs. A job title such as `Manager` is supporting evidence only; it cannot alone establish `advanced`.

### 5.4 Impact signals

- explicit scale: team size, portfolio, budget, volume, geographic breadth, or user/customer count;
- explicit result: revenue, cost, quality, time, risk, retention, delivery, or service improvement;
- explicit quantified improvement: a truthful number with an outcome context.

Numbers without outcome context are not impact. A single metric can support experienced evidence but does not by itself prove advanced stage.

### 5.5 Deterministic evaluation and confidence

The future internal assessor should record canonical evidence and calculate stage evidence tiers rather than expose a public score:

1. detect entry, ownership, complexity, cross-functional, people-leadership, and impact signals;
2. determine whether evidence supports entry, developing, experienced, or advanced according to the taxonomy;
3. choose the highest stage with at least two independent evidence categories, except `entry`, which may use credible project/internship evidence plus foundational capability;
4. return `unknown` when no stage has sufficient evidence or signals conflict without a clear lead;
5. assign internal confidence: `high` for three or more independent categories including responsibility or impact, `medium` for two, `low` for one or for `unknown`.

Stage ranking is `entry < developing < experienced < advanced`. Evidence of people management or broad organisational ownership may elevate a stage when corroborated by a second category. Long tenure alone cannot elevate a stage beyond `developing`; repeated role entries alone cannot elevate a stage beyond `experienced`.

## 6. Internal and public output contract

### 6.1 Internal schema

```python
class CareerStage(str, Enum):
    ENTRY = "entry"
    DEVELOPING = "developing"
    EXPERIENCED = "experienced"
    ADVANCED = "advanced"
    UNKNOWN = "unknown"

@dataclass(frozen=True, slots=True)
class CareerStageAssessment:
    stage: CareerStage
    confidence: StageConfidence
    evidence: tuple[str, ...]
```

`StageConfidence` is internal `high | medium | low`. It expresses evidence sufficiency, not candidate quality. These types remain under `skill/career/` and are not exported through `skill.schemas`, Agent schemas, `NextMoveSkill`, Application contracts, or `JobMatchResult`.

### 6.2 Legacy mapping

Public `ResumeAnalysisResult.career_level` retains its current type, field name, default, and Agent enum. The deterministic compatibility mapping is:

| Internal stage | Public `career_level` / `current_level` |
|---|---|
| `entry` | `junior` |
| `developing` | `mid` |
| `experienced` | `senior` |
| `advanced` | `lead` |
| `unknown` | `unknown` |

`intern` remains an accepted legacy enum value but is not newly emitted in v1. This follows the existing analyzer, which currently never emits it. `CareerAdvisor` uses the internal assessment to select advice, while it continues returning only the mapped `current_level` string.

No public field is added, removed, renamed, made nullable, or retyped. `NextMoveSkill.run("analyze_resume", ...)`, `NextMoveSkill.run("career_advice", ...)`, and `NextMoveSkill.run("career_analysis", ...)` retain inputs, capability names, sequence, and response shapes. The Agent `RESUME_ANALYSIS_OUTPUT_SCHEMA` and `CAREER_ADVICE_OUTPUT_SCHEMA` are unchanged.

## 7. Career Advice integration

The assessor is computed once during analysis. `CareerAdvisor.advise()` uses the supplied analysis stage when available; if analysis is absent, it computes the same internal assessment from the profile. It must not use an Agent-only or Application-only data source.

| Stage | Possible-path emphasis | Skill/advice emphasis | Next action emphasis |
|---|---|---|---|
| `entry` | internships, junior/coordinator/assistant roles, structured entry paths | foundational skills and visible work samples | build a focused project/portfolio, seek supervised practice, describe learning evidence truthfully |
| `developing` | specialist and independently accountable roles | depth in a target domain, end-to-end ownership, stakeholder communication | document ownership, complete a scoped improvement, target a next-level role with adjacent scope |
| `experienced` | senior specialist, lead contributor, functional manager where evidence supports it | systems/process depth, cross-functional influence, mentoring | quantify impact, demonstrate a complex initiative, develop leadership or strategic communication evidence |
| `advanced` | principal, lead, manager, head-of-function paths according to evidence | strategy, organisational design, talent development, portfolio/business impact | clarify target scope, evidence team/organisational outcomes, build succession and strategic narrative |
| `unknown` | general exploration only | fill missing resume evidence before a stage claim | add truthful role/project/responsibility detail and choose a target direction |

Career changers keep the highest transferable internal stage that evidence supports. Advice must distinguish transferable evidence from target-domain gaps: it can recommend a bridge project, role-specific learning, or a scoped transition, but must not call the user entry-level solely due to a new domain.

## 8. Testing strategy

Implementation follows TDD across internal assessment, analyzer mapping, advice behavior, and public contract regression.

1. **New graduate:** internship/project/foundational evidence -> internal `entry`, mapped public `junior`, entry actions; no fabricated experience.
2. **2-3 year professional:** dated or explicit tenure plus independent module/process ownership -> `developing`, mapped `mid`, ownership-oriented actions.
3. **5+ year expert:** complex work, cross-functional influence, and quantified improvement -> `experienced`, mapped `senior`, impact/leadership actions even without direct reports.
4. **Management role:** explicit people leadership plus process/portfolio/business evidence -> `advanced`, mapped `lead`, management/strategy actions. A manager title alone must not meet this case.
5. **Career changer:** demonstrated senior transferable responsibility with a new-domain target profile -> retained transferable stage and bridge advice, not automatic entry downgrade.

Also cover empty/insufficient profile -> `unknown`, title-only inflation prevention, duration-only cap, contradictory evidence, Chinese/English aliases, date parsing, deterministic evidence ordering, no contact-data use, and serialization/Agent schema exactness.

The full existing test suite must continue to pass. Explicit legacy fixtures should be updated only where a new mapped value is an intentional, documented result of richer evidence; contract keys and permitted enum values remain exact.

## 9. Implementation boundaries

The future implementation may modify focused files under `skill/career/`, `skill/analysis/`, their internal schemas, and focused analysis/career/serialization/Agent regression tests. It may update `CHANGELOG.md`, `PROJECT.md`, and `NEXT_TASK.md` only after acceptance.

It must not modify `skill/interface.py`, `skill/agent/`, `skill/matching/`, `skill/ai/application/`, Application Layer, backend, frontend, CLI, provider code, dependency manifests, release files, or Git tags.

## 10. Design acceptance criteria

The design is frozen when the four stages and `unknown` are explicitly defined; duration, responsibility, complexity, management, and impact are evaluated without years-only classification; internal assessment and legacy mapping preserve public contracts; career advice behavior is stage-specific and career-change-safe; and new graduate, developing professional, expert, manager, career changer, and insufficient-information tests are specified.
