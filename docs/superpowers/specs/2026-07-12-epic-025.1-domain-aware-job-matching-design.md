# Epic-025.1 Domain-aware Job Matching Design

## 1. Decision and scope

Epic-025.1 freezes a deterministic, provider-neutral design that upgrades job matching from one global technical-keyword list to career-domain-aware matching. The implementation remains inside `skill/matching/`; `NextMoveSkill.match_job()` and `NextMoveSkill.run()` keep their current signatures and behavior boundary.

This Epic is design-only. It does not modify runtime code, the Skill API, Application Layer, Agent Contract, frontend, backend, providers, databases, dependencies, or the `v0.8.0` tag. The dependency direction remains application -> public Skill contract; `skill/` must never import an Application Layer module.

## 2. Current architecture and failure analysis

### 2.1 Module boundary

- `skill/matching/matcher.py` owns `JobMatcher`, keyword extraction, scoring, and user-facing explanations.
- `skill/matching/schemas.py` owns the provider-neutral `JobMatchResult` dataclass.
- `skill/matching/__init__.py` exports `JobMatcher` and `JobMatchResult`.
- `skill/schemas/matching.py` is a compatibility import for `JobMatchResult`.
- `skill/interface.py` accepts `ResumeProfile | str` and a `job_description: str`, parses raw resumes, and delegates to `JobMatcher.match(profile, job_description)`.
- `skill/agent/tools.py` separately freezes the Agent-visible JSON output schema.
- `tests/test_job_matcher.py` covers a strong technical match, missing technical keywords, and an empty resume. Interface, API, Agent, workflow, Pilot, serialization, and acceptance tests also consume `JobMatchResult`.

### 2.2 Current input and output

The matcher input is a structured `ResumeProfile` plus an unstructured job-description string. Available resume evidence includes summary, roles and highlights, skills, education, projects, certifications, languages, and raw text, although the matcher currently uses only `profile.skills` and whether `profile.experience` is non-empty.

The public output is:

```python
JobMatchResult(
    match_score: int,
    matched_skills: list[str],
    missing_skills: list[str],
    strengths: list[str],
    gaps: list[str],
    recommendations: list[str],
)
```

### 2.3 Current score generation

`KNOWN_JOB_KEYWORDS` recognizes 16 normalized terms, dominated by English software and data technologies. `_extract_job_keywords()` tokenizes only Latin-alphabet terms. The score is:

```text
if resume has neither skills nor experience: 0
if job description contains no known keyword: 0
otherwise: round(matched known keywords / job known keywords * 80)
           + 20 when any experience entry exists
           capped at 100
```

The experience bonus measures presence, not relevance. HR, administration, finance, operations, manufacturing, supply-chain, sales, marketing, customer-service, and Chinese-language evidence can therefore be invisible. A result can report `match_score=0` even when the resume and role are strongly aligned.

### 2.4 Compatibility risks

`JobMatchResult` is imported through two public paths, serialized recursively, embedded in `CareerAnalysisReport`, and described by a separate Agent JSON Schema. Adding dataclass fields would make serialized Skill results diverge from the frozen Agent output contract unless that contract also changed. Renaming or removing the six current fields would break direct consumers and tests. Changing technical-role scores can also break threshold assertions and Pilot fixtures. These risks require additive internal contracts, unchanged public fields, and regression fixtures for legacy technical cases.

## 3. Alternatives considered

### A. Expand the global keyword list

This is the smallest code change but retains one undifferentiated vocabulary. Ambiguous terms, missing-domain penalties, and explanations become increasingly unreliable as the list grows. Rejected as the target architecture.

### B. Deterministic domain-aware pipeline (selected)

Classify resume and job evidence into a small taxonomy, select domain/job-family vocabularies, and calculate domain, skill, and qualification components. This remains offline, explainable, dependency-free, and straightforward to test. Vocabulary quality can improve without changing public APIs.

### C. AI semantic classification and scoring

Model reasoning could recognize broader phrasing, but it introduces provider ownership, nondeterminism, cost, privacy, and availability concerns. It violates this Epic's constraints and is deferred.

## 4. Target architecture

The future implementation should keep `JobMatcher` as the single public matching facade and compose focused internal units:

```text
ResumeProfile + job_description
        -> evidence normalization
        -> resume domain classification
        -> job domain/job-family classification
        -> domain-specific skill extraction
        -> qualification extraction
        -> weighted scoring and confidence
        -> explanation mapping
        -> unchanged JobMatchResult
```

Proposed files:

- `skill/matching/taxonomy.py`: immutable domain, job-family, alias, and evidence dictionaries.
- `skill/matching/classifier.py`: deterministic classification and evidence reporting.
- `skill/matching/scoring.py`: component scores, confidence, and bounded aggregation.
- `skill/matching/schemas.py`: current public result plus internal matching dataclasses that are not exported through the Agent contract.
- `skill/matching/matcher.py`: orchestration and compatibility mapping only.

No unit may import from `skill.ai`, an Application Layer, backend, or frontend. Taxonomy data is static Python data in v1; no database or new package is needed.

## 5. Domain Classification

### 5.1 Career Domain Taxonomy v1

`CareerDomain` contains these stable machine values:

- `human_resources`
- `technology`
- `finance`
- `sales`
- `marketing`
- `operations`
- `supply_chain`
- `manufacturing`
- `customer_service`
- `other`
- `unknown`

`other` means there is positive occupational evidence but it does not fit a supported v1 domain. `unknown` means there is insufficient or conflicting evidence to classify safely. They are not synonyms.

### 5.2 Classification inputs

Job classification consumes the complete `job_description`. Resume classification consumes all non-empty professional evidence available in `ResumeProfile`: `summary`, experience roles and highlights, `skills`, education fields and descriptions, project names/descriptions/technologies, certifications, and `raw_text`. Personal contact information is excluded.

Normalization is deterministic and multilingual-ready: Unicode case normalization, punctuation/whitespace normalization, exact phrase aliases, and bounded token/phrase matching. V1 dictionaries should include focused English and Chinese aliases for domain titles, job-family titles, skills, responsibilities, and qualifications. Substring matches inside unrelated words are prohibited.

### 5.3 Recognition rule

Each evidence hit has a fixed weight by signal quality:

- exact job-family or role-title phrase: 4;
- domain-specific responsibility or certification: 3;
- domain-specific skill phrase: 2;
- broad supporting term: 1.

Scores are summed per domain after alias deduplication. Classify the highest domain only when its score is at least 4 and exceeds the runner-up by at least 2. When positive occupational evidence exists but no supported domain reaches 4, return `other`. When no occupational evidence exists, or the top two supported domains differ by less than 2, return `unknown`. The winning job family uses the same evidence but requires at least one exact family/title or two family-specific signals; otherwise it is `None`.

### 5.4 Internal output schema

```python
@dataclass(frozen=True, slots=True)
class DomainClassification:
    domain: CareerDomain
    job_family: JobFamily | None
    confidence: MatchConfidence
    evidence: tuple[str, ...]
```

`MatchConfidence` is `high | medium | low`. Classification confidence is high when the top score is at least 8 and leads by at least 4, medium when the classification thresholds are met, and low for `other` or `unknown`. Evidence contains canonical, non-sensitive phrases actually found in input and is never fabricated.

## 6. Job Family v1

The first release deliberately keeps families small. Families are subordinate to one domain and are not a new public API field.

| Domain | Job families |
|---|---|
| `human_resources` | `hr_generalist`, `recruitment`, `compensation_benefits`, `employee_relations` |
| `technology` | `backend`, `frontend`, `data`, `devops` |
| `finance` | `accounting`, `financial_analysis`, `audit_tax` |
| `sales` | `business_development`, `account_management`, `sales_operations` |
| `marketing` | `content_brand`, `digital_marketing`, `market_research` |
| `operations` | `business_operations`, `project_operations`, `administration` |
| `supply_chain` | `procurement`, `logistics`, `planning` |
| `manufacturing` | `production`, `quality`, `maintenance` |
| `customer_service` | `customer_support`, `customer_success`, `service_operations` |
| `other`, `unknown` | no v1 family |

Cross-functional roles may generate evidence in more than one domain, but v1 returns one primary domain. The explanation may acknowledge a close secondary domain without blending its vocabulary into missing-skill penalties.

## 7. Matching Model

### 7.1 Components and weights

The overall score remains an integer from 0 to 100:

```text
match_score = round(
    domain_match * 0.30
  + skill_match * 0.50
  + qualification_match * 0.20
)
```

Every component is in `0..100` and the aggregate is clamped to `0..100`.

#### Domain Match (30%)

- 100: same known domain and same known job family;
- 90: same known domain and either family is unavailable;
- 75: same known domain but different known families;
- 55: adjacent-domain mapping explicitly listed in taxonomy, such as operations/supply-chain;
- 30: one side is `other` or `unknown` and there is some transferable evidence;
- 10: known domains conflict;
- 0: there is no resume occupational evidence.

Domain mismatch is a signal, not an automatic zero. Adjacent-domain mappings must be explicit and symmetric.

#### Skill Match (50%)

Extract required job skills only from the selected domain and job-family vocabulary plus a small neutral vocabulary such as communication, Excel, leadership, and project management. Each requirement is marked `required` or `preferred` only from explicit job-description language; required weight is 2 and preferred/default weight is 1. Canonical aliases are deduplicated.

```text
skill_match = matched requirement weight / total requirement weight * 100
```

Resume evidence may come from skills, summaries, roles, highlights, projects, certifications, or raw text. A job requirement is never inferred solely because it exists in the taxonomy. When the job contains no extractable skill requirement, the component is unavailable rather than zero and its weight is redistributed proportionally across available components.

#### Qualification Match (20%)

V1 recognizes only explicit, conservatively comparable requirements: years of experience, education level/field, named certifications, and language requirements. Satisfied weight / explicit requirement weight produces the component. Missing resume evidence remains a gap; ambiguous dates or qualifications are not guessed. If the job states no qualification requirement, this component is unavailable and its weight is redistributed.

At least one score component must be available. If the job description yields no classifiable domain, skills, or qualifications, return `match_score=0`, low confidence, and a gap explaining insufficient job information.

### 7.2 Overall confidence

The internal `MatchAssessment` carries `high | medium | low` confidence:

- `high`: job domain is known with high classification confidence, at least three weighted requirements are extracted, and at least two components are available;
- `medium`: job domain is known, at least one requirement is extracted, and at least two components are available;
- `low`: job domain is `other/unknown`, only one component is available, no requirements are extractable, or classification is ambiguous.

Confidence describes evidence sufficiency, not how high the score is. A low score can have high confidence.

### 7.3 Internal scoring schema

```python
@dataclass(frozen=True, slots=True)
class MatchAssessment:
    score: int
    confidence: MatchConfidence
    domain_score: int | None
    skill_score: int | None
    qualification_score: int | None
    strengths: tuple[str, ...]
    gaps: tuple[str, ...]
```

Example internal assessment:

```json
{
  "score": 87,
  "confidence": "high",
  "domain_score": 100,
  "skill_score": 82,
  "qualification_score": 85,
  "strengths": ["同属 human_resources / recruitment 岗位族", "已覆盖招聘流程与面试协调"],
  "gaps": ["职位要求劳动法规经验，简历中未见明确证据"]
}
```

### 7.4 Explanation mapping

Public output remains explainable through existing fields:

- `match_score` receives `MatchAssessment.score`;
- `matched_skills` lists canonical job requirements with resume evidence;
- `missing_skills` lists explicit job requirements without resume evidence;
- `strengths` begins with domain/job-family alignment when supported, then skill and qualification evidence;
- `gaps` identifies domain ambiguity, explicit missing requirements, and insufficient evidence without claiming the candidate lacks an ability;
- `recommendations` gives truthful evidence-gathering or resume-tailoring actions and never advises adding unsupported claims.

Explanations should name the component and evidence, for example: `领域匹配：简历与岗位均显示 human_resources / recruitment 信号。` When confidence is low, recommendations must ask for more job or resume detail rather than present a precise score as authoritative.

## 8. Backward Compatibility

### 8.1 Frozen public contract

`JobMatcher.match(profile, job_description) -> JobMatchResult`, `NextMoveSkill.match_job()`, and `NextMoveSkill.run()` remain unchanged. The six existing `JobMatchResult` fields retain names, types, defaults, and meaning. No existing field becomes optional or nullable.

This implementation slice will not add `domain`, `job_family`, `confidence`, or component scores to `JobMatchResult`. Although optional fields are source-compatible for many Python callers, dataclass serialization would expose them and change the Agent-visible payload without changing `JOB_MATCH_OUTPUT_SCHEMA`. Such additions require a separate, explicitly approved Agent Contract versioning Epic.

The new domain/classification/assessment dataclasses are internal to `skill.matching` and are not exported from `skill.schemas`, Agent Tool schemas, `NextMoveSkill`, or Application Layer contracts.

### 8.2 Behavioral compatibility

Scores are quality behavior and may improve, but legacy technical fixtures must retain their semantic bands: the current complete backend fixture remains at least 80, the partial data fixture remains below 80 with the same visible missing skills, and an empty resume remains 0. Output list ordering is deterministic. Existing imports and serialization continue to work.

Old Agent calls are unaffected because inputs, capability names, required fields, output field names, and JSON Schema remain unchanged. Agents gain better field content but no new contract requirement.

## 9. Testing Strategy

Implementation uses TDD at the internal classifier, scorer, matcher, and public regression boundaries.

### 9.1 Domain fixtures

1. HR: recruitment/generalist resume against an HR role; domain is `human_resources`, real HR requirements match, score is non-zero and explained without technical keywords.
2. Technology: preserve the existing backend fixture and missing Docker/AWS behavior.
3. Finance: accounting or financial-analysis evidence matches reconciliation, reporting, Excel, or certification requirements without requiring software-stack terms.
4. Operations: administration/business-operations evidence matches coordination, process, vendor, or project-operations requirements.
5. Insufficient information: vague job text produces `unknown`, score 0, low confidence internally, and a clear public gap/recommendation requesting details.

Add focused coverage for Chinese and English aliases, ambiguous cross-domain text, `other` versus `unknown`, family classification, required-versus-preferred weighting, qualification comparisons, weight redistribution, score clamping, stable ordering, and no fabricated missing requirement.

### 9.2 Compatibility suite

- Keep and run all current `tests/test_job_matcher.py` assertions.
- Verify `skill.schemas.matching.JobMatchResult` remains the same class and serializes to exactly the six established keys.
- Verify `JOB_MATCH_OUTPUT_SCHEMA` and Agent Tool inputs/outputs are byte-for-byte unchanged for this Epic.
- Run interface, API, Agent runtime, workflow, Pilot, release acceptance, and full non-live unit tests.
- Run an architecture test ensuring `skill/` does not import Application Layer packages and the existing application -> skill direction remains intact.

## 10. Error and edge-case policy

- Empty resume: score 0; extracted job requirements remain visible as gaps; explanation says resume evidence is insufficient.
- Empty or uninformative job description: score 0; no invented missing skills; ask for responsibilities and qualifications.
- Mixed-domain role: select a primary domain only when the lead threshold is met; otherwise `unknown` with low confidence.
- Unsupported recognized occupation: `other`, not `technology` by default.
- Duplicate aliases: one canonical requirement and one contribution to score.
- Negated terms such as `no certification required`: do not create a requirement.
- Missing evidence means “not visible in the resume,” never “candidate does not possess it.”

## 11. Implementation boundaries

The approved future implementation may modify only `skill/matching/`, focused matching/schema/architecture tests, and Epic maintenance documents. It must not modify:

- `skill/interface.py` or any public `NextMoveSkill` entrypoint;
- `skill/agent/` or Agent schemas/contracts;
- `skill/ai/application/` or any Application Layer;
- backend, frontend, Web, provider, database, CLI, packaging, or dependency files;
- the `v0.8.0` tag.

## 12. Design acceptance criteria

The design is frozen when the taxonomy distinguishes supported domains from `other` and `unknown`; job families remain intentionally small; classification inputs, thresholds, schemas, scoring weights, confidence, and explanations are deterministic; public compatibility is explicit; HR, technology, finance, operations, and insufficient-information tests are specified; and implementation remains inside the Skill Core matching boundary without changing API or Agent contracts.
