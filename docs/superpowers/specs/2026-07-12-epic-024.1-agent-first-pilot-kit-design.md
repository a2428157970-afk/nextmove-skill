# Epic-024.1 Agent-first Pilot Kit Design

## 1. Goal

Epic-024 validates whether NextMove Skill helps people understand their career situation and choose useful next actions. The first Pilot audience is non-technical job seekers using NextMove through an Agent.

The first-stage success condition is not only that the program runs. A Pilot succeeds when a user completes a career analysis, understands the report, and considers at least one recommendation actionable.

## 2. Scope

Epic-024.1 freezes the Pilot experience specification for:

1. the Quick Start user journey;
2. fictional sample resume and job-description data;
3. reusable Agent Prompt templates;
4. the output-understanding guide;
5. the dual-entry feedback boundary.

This design does not implement product behavior. It does not add a Skill capability, change `NextMoveSkill`, change Agent Tool schemas, change `career_analysis`, or modify the CLI. It adds no Web application, backend, database, authentication, AI Provider, telemetry SDK, or large architecture change. Git tag, push, and GitHub Release work remains paused.

## 3. Primary user and value hypothesis

### Primary user

A non-technical job seeker who can paste instructions and career information into an Agent but does not need to understand Python, JSON schemas, command-line packaging, or repository internals.

### Value hypothesis

Given a truthful resume and an optional target role, the user can use NextMove through an Agent to identify:

- the strongest and weakest resume signals;
- truthful resume improvements;
- job fit and visible skill gaps;
- a small set of practical next actions.

The Agent must preserve the boundary between facts supplied by the user and suggestions generated by NextMove. It must not invent employment, skills, education, projects, certifications, achievements, or metrics.

## 4. Quick Start user journey

The primary journey is Agent-first and should take no more than five minutes with the fictional sample:

```text
Open Quick Start
    -> choose an Agent Prompt
    -> use the fictional sample or paste a privacy-reviewed resume
    -> optionally add a target job description
    -> ask the Agent to invoke NextMove
    -> receive a four-part career analysis
    -> read the short interpretation guide
    -> copy the anonymous feedback template
```

The Quick Start must lead with the Agent path. CLI and Python paths remain secondary validation paths for developers and advanced users.

### Quick Start content order

1. What NextMove can help with.
2. Privacy and non-fabrication warning.
3. Fastest sample-based Agent experience.
4. Try with the user's own privacy-reviewed input.
5. How to read the four report sections.
6. How to send anonymous feedback.
7. Links to CLI and Python instructions.

### Quick Start completion definition

A user has completed the journey when the Agent returns a successful report containing `analysis`, `improvement`, `job_match`, and `career_advice`. A clear structured failure is an understandable error outcome but does not count as completion.

## 5. Sample data design

### Files

The implementation phase should add:

```text
examples/data/sample_resume.txt
examples/data/sample_job_description.txt
```

### Sample resume requirements

The sample resume must be entirely fictional and contain no real person's identity or contact information. It should describe a plausible early-career candidate with enough evidence to produce useful analysis but enough gaps to make the improvement and matching sections meaningful.

It should include:

- a fictional professional summary;
- one or two fictional roles;
- a short skills section;
- at least one quantified achievement clearly labeled as fictional sample data;
- one project;
- no email, phone number, street address, account identifier, or real employer name.

### Sample job-description requirements

The job description must be fictional and concise. It should overlap with some resume skills and include two or three visible gaps so the matching output demonstrates both strengths and missing skills.

It must not copy a real job posting or identify a real employer.

### Sample acceptance criteria

- Both files are valid UTF-8 text.
- CLI and Agent examples can consume them without editing.
- The sample produces a successful `career_analysis` response offline.
- Expected output contains non-empty analysis, improvement, job-match, and career-advice sections.
- No sample contains personal data, company-confidential information, or credentials.

## 6. Agent Prompt template structure

The Pilot Kit should provide reusable prompts for:

1. complete career analysis;
2. resume improvement;
3. target-job matching;
4. career direction and next actions.

Each prompt must contain the following blocks:

### Role and invocation

- State that the Agent should use NextMove Skill.
- Name the requested capability or outcome.
- Tell the Agent to report a structured failure rather than silently improvising when required input is missing.

### User input placeholders

- `RESUME_TEXT`
- `JOB_DESCRIPTION` when matching or complete analysis is requested.

The template must explain that `job_description` is required for Agent `match_job` and `career_analysis` calls. It may be omitted only for focused capabilities that do not require it.

### Truthfulness rules

- Use only facts present in the supplied input.
- Do not invent experiences, skills, education, projects, certifications, metrics, employers, or credentials.
- Mark suggestions as suggestions, not as existing facts.
- State when the available information is insufficient.

### Response presentation

For complete analysis, present:

1. resume strengths and weaknesses;
2. top improvement priorities;
3. job match and visible skill gaps;
4. three practical next actions.

The Agent may summarize the structured report for readability but must not alter scores, add unsupported facts, or hide a Skill failure.

## 7. Output-understanding guide

The guide should translate `CareerAnalysisReport` into user language without changing its contract.

| Report section | User question | Reading guidance |
|---|---|---|
| `analysis` | What does my resume currently communicate? | Separate strengths from weak or missing evidence. Career level is a rule-based signal, not a final judgment. |
| `improvement` | What should I change first? | Prioritize truthful, evidence-backed improvements. Do not add a claim merely to satisfy a suggestion. |
| `job_match` | How well does the resume support this target role? | Treat the score as directional. Check matched and missing skills against the actual resume and role. |
| `career_advice` | What should I do next? | Select a small number of practical actions and validate major career decisions with human judgment. |

### Human-readable summary

The Pilot documentation should show a five-part view:

```text
Top strengths
Top three improvements
Job match
Visible skill gaps
Next three actions
```

This is a presentation guide only. Default CLI JSON and the `CareerAnalysisReport` schema remain unchanged in Epic-024.1.

### Error interpretation

The guide must explain that a structured failure is useful feedback. Missing resume text, a missing Agent `job_description`, unreadable files, and unsupported capability names should not be converted into invented analysis.

## 8. Feedback boundary

Feedback uses two entrypoints:

```text
Pilot user
    |-- copyable anonymous feedback template (primary)
    `-- GitHub Issue template (developer and advanced-user channel)
```

### 8.1 Primary anonymous feedback template

Create:

```text
docs/pilot-feedback-template.md
```

The template must be platform-neutral and copyable into an Agent conversation, community post, email, or Issue. It should collect:

- usage path: ChatGPT Agent, Claude, Cursor, Codex, CLI, or Python API;
- scenario: resume analysis, job matching, resume improvement, or career planning;
- whether first use completed successfully;
- approximate completion time;
- most valuable report section;
- output-understanding rating from 1 to 5;
- observed error category;
- free-form improvement suggestion.

### 8.2 GitHub Pilot Issue template

Create:

```text
.github/ISSUE_TEMPLATE/pilot_feedback.md
```

This secondary channel is for developers and advanced users reporting installation failures, CLI bugs, Skill invocation failures, output-contract problems, or detailed Pilot observations. It should use the same privacy boundary as the anonymous template and avoid requiring a full resume or job description for reproduction.

### 8.3 Pilot guidelines

Create:

```text
docs/pilot-guidelines.md
```

The guidelines should explain Pilot purpose, supported usage paths, truthful input expectations, feedback channels, success metrics, and the data boundary.

## 9. Pilot data boundary

### Allowed collection

- usage path;
- scenario category;
- completion success;
- approximate completion time;
- most valuable report section;
- output-understanding rating;
- general error category;
- non-sensitive improvement suggestion.

### Prohibited collection

- resume or job-description body;
- real name;
- phone number;
- email or other contact detail;
- employer or company-confidential information;
- API keys, tokens, cookies, passwords, or other credentials;
- raw Agent conversation containing private career information.

All templates must place the privacy warning before free-text fields. Users should describe a problem with an anonymous summary or fictional reproduction.

No analytics service, database, hidden telemetry, or automatic content collection is introduced. Pilot metrics are calculated only from voluntarily submitted, privacy-reviewed feedback.

## 10. Pilot metrics

| Metric | First-stage target | Definition |
|---|---:|---|
| First-use completion | >= 80% | User receives a successful report containing all four career-analysis sections. |
| Report understanding | >= 80% | User rates understanding 4 or 5, or explicitly states the report was easy to understand. |
| Recommendation actionability | >= 70% | User identifies at least one recommendation they can act on. |
| Severe factual errors | 0 | No supported feedback identifies invented experience, skill, credential, employer, or metric as a user fact. |
| Feedback completion | >= 30% | At least 30% of invited Pilot participants submit the short feedback template. |

Metrics must be accompanied by the number of Pilot participants. Small samples are directional evidence, not statistical proof.

## 11. Epic structure and sequencing

```text
024.1 Quick Adoption Kit
    -> 024.2 Agent Prompt Templates
    -> 024.3 Output Understanding Guide
    -> 024.4 Pilot Feedback System
    -> 024.5 Pilot Review
```

The implementation sequence should deliver one usable path as early as possible:

1. add fictional sample files and the Agent-first Quick Start;
2. add the reusable Agent prompts;
3. add the output-reading guide;
4. add the anonymous and GitHub feedback templates plus Pilot guidelines;
5. run repository acceptance checks;
6. invite a small Pilot and review real feedback before changing CLI behavior.

## 12. Deferred decisions

The following decisions require real Pilot evidence and are not part of this design implementation:

- adding `--job-description-file`;
- adding a human-readable CLI summary mode;
- changing CLI error formatting;
- adding automated feedback collection;
- changing Skill rules, schemas, capabilities, or providers.

## 13. Validation requirements for implementation

The future implementation plan must verify:

- every Quick Start command uses a real repository file and runs successfully;
- the complete Agent Prompt includes required inputs and non-fabrication rules;
- sample data is fictional and contains no contact details or credentials;
- the default CLI JSON and Skill API behavior are unchanged;
- the full existing unit suite and compilation checks pass;
- `git diff --check` passes;
- only documentation, fictional sample data, issue templates, and focused adoption acceptance tests are changed unless separately approved.

## 14. Acceptance criteria

Epic-024.1 design is frozen when:

- Agent-first is explicitly the primary Pilot path;
- the five-minute user journey is defined;
- sample data requirements are safe and testable;
- Prompt templates preserve required input and truthfulness boundaries;
- the output guide maps all four report sections to user questions;
- anonymous feedback is the primary entrypoint and GitHub Issue is secondary;
- collected and prohibited data are explicit;
- success metrics measure completion, understanding, actionability, factual safety, and feedback participation;
- CLI and product behavior changes remain deferred pending Pilot evidence.
