# Epic-024.3 Pilot Execution Design

## 1. Goal

Move NextMove from Agent-first Pilot Kit Ready into real-user validation.

The central question is:

> Can a non-technical job seeker independently complete a career analysis through an Agent and consider the output practically valuable?

Epic-024.3 discovers adoption, value, and quality problems. It does not pursue growth, statistical significance, or new product infrastructure.

## 2. Pilot positioning

### Primary audience

The first-round primary audience is non-technical job seekers using:

```text
ChatGPT / Claude / Cursor / Codex
    -> NextMove Skill
    -> CareerAnalysisReport
```

Primary measurements come only from this Agent-first audience.

### Secondary audience

Developers may validate the Python API, CLI, and Skill integration. Their observations are useful technical feedback but must be reported separately and excluded from first-round activation and value rates.

## 3. Pilot size and composition

The first round targets 5-10 completed participant attempts. The purpose is to identify recurring friction and value signals, not to measure market demand.

Recommended composition:

| User type | Target count | Main validation question |
|---|---:|---|
| Graduate / early career | 2-3 | Can the user understand the basic analysis and improvements? |
| 3-5 years of experience | 2-3 | Does job matching reveal useful evidence and gaps? |
| Career transition | 1-2 | Is career advice specific enough to guide next actions? |
| Manager / senior professional | 1-2 | Can the workflow handle a more complex background without misleading simplification? |

The composition is a recruitment target, not a reason to collect employer, identity, or detailed career-history data. Record only one broad user-type category.

## 4. Pilot materials

Each participant receives only:

- `docs/quick-start.md`;
- `docs/agent-prompt-template.md`;
- the fictional sample files as an optional rehearsal;
- `docs/output-guide.md`;
- `docs/pilot-feedback-template.md` after the attempt.

The recruitment message must explain that NextMove is an experimental Developer Preview, not a hiring service or guaranteed career recommendation. It must instruct participants to remove identifying and confidential information before using any Agent.

## 5. Pilot flow

### Step 1: Invitation and consent boundary

Send the recruitment message and Pilot materials. Participation and feedback are voluntary. The Pilot operator must not request a resume, job description, contact details inside the Pilot record, Agent transcript, screenshots containing private data, or credentials.

### Step 2: Independent first attempt

The participant should start with the Quick Start and complete the first attempt without live coaching. The operator may observe only whether the participant knows the next step, successfully invokes NextMove, and understands the result.

If the participant becomes blocked, record the attempt as not independently completed before providing minimal assistance. The participant may continue after assistance, but assisted completion and independent completion must remain separate fields.

### Step 3: Privacy-safe feedback

After the attempt, the participant completes `docs/pilot-feedback-template.md`. The template must collect two additional fields required by this design:

- whether the analysis was completed without human help;
- whether the participant would use NextMove again.

Feedback must use categories and anonymous summaries. It must not contain resume or job-description bodies.

### Step 4: Manual classification

Classify every privacy-reviewed record as:

- activation outcome;
- value outcome;
- quality observation;
- one or more feedback categories from Section 9.

Classification must not reinterpret a participant's answer to make a metric look better. Ambiguous answers remain `unknown` and are excluded from that metric's denominator.

### Step 5: Review gate

Generate the Pilot Review Report only after 5-10 eligible primary-audience attempts have been recorded or the operator explicitly stops the round for a severe safety problem. Do not claim that the Pilot is complete merely because templates exist.

## 6. Pilot record boundary

The execution phase should create a copyable, content-blind record template. One record represents one attempt and uses a local anonymous sequence such as `pilot-001`; it must not contain a participant identifier that can be mapped back to a person.

Allowed fields:

```text
anonymous attempt id
broad user type
Agent type / usage path
scenario category
analysis completed: yes / no
completed without human help: yes / no / unknown
completion time range
output understanding: 1-5
recommendation helpful: yes / no / unknown
would use again: yes / no / unknown
most valuable section
problem categories
severe factual error: yes / no / unknown
privacy-safe summary
```

Records are manually created from voluntary feedback. Epic-024.3 introduces no database, analytics platform, telemetry, automated logging, or raw-content retention.

## 7. Data boundary

### Allowed collection

- broad user-type category;
- usage path and Agent type;
- scenario category;
- completion and independent-completion outcomes;
- completion-time range;
- experience and understanding ratings;
- most valuable section;
- helpfulness and reuse-intent answers;
- problem and suggestion categories;
- privacy-safe anonymous summary.

### Prohibited collection

- full or partial resume body;
- full or partial job-description body;
- real name or persistent participant handle;
- phone number, email, address, or contact information;
- employer identity or company-confidential information;
- API key, token, password, cookie, credential, or authentication artifact;
- raw Agent conversation, unredacted screenshot, prompt transcript, or model response containing private career information.

If prohibited data is submitted, do not copy it into the Pilot record or Review Report. Ask the participant to resubmit an anonymous category-level description, and use only that replacement.

## 8. Metrics and calculation rules

Primary metrics use eligible non-technical Agent-first attempts only. Always publish the numerator, denominator, and rate; do not report a percentage without the sample count.

### Activation

| Metric | Target | Calculation |
|---|---:|---|
| First analysis completed | >= 80% | completed attempts / all eligible attempts |
| Completed without human help | >= 60% | independently completed attempts / all eligible attempts |
| Output understood | >= 80% | understanding score 4 or 5 / attempts with a rating |

### Value

| Metric | Target | Calculation |
|---|---:|---|
| Recommendation considered helpful | >= 70% | `yes` / attempts answering helpfulness |
| Willing to use again | >= 60% | `yes` / attempts answering reuse intent |
| Severe factual errors | 0 | count of confirmed severe factual errors |

### Quality observations

Review every record for:

- invented experience, skill, education, project, employer, credential, achievement, or metric;
- incorrect or unsupported skill interpretation;
- advice that is too generic to act on;
- advice that is infeasible, unsafe, or presented as a guaranteed outcome;
- mismatch between the structured report and the Agent's user-facing summary.

A severe factual error means NextMove or the Agent presents an unsupported career fact as if it came from the participant. A suggestion clearly labeled as a future option is not a factual error.

## 9. Feedback classification

### A. Adoption and usage issue

Examples:

- the user does not know how to start;
- the Prompt is unclear;
- the Agent cannot discover or invoke NextMove;
- the user cannot tell whether analysis completed.

Disposition: consider for UX and adoption-document improvements.

### B. Output and quality issue

Examples:

- inaccurate analysis;
- generic or non-actionable advice;
- weak job matching;
- incorrect skill gap;
- unsupported factual claim.

Disposition: consider for Quality Improvement after reproducing with fictional or synthetic inputs.

### C. Product request

Examples:

- save or share a report;
- track career changes;
- add a new delivery surface;
- automate follow-up actions.

Disposition: record only. Do not implement during Epic-024.3.

One record may have multiple categories. Product requests must not be treated as validated demand solely because one participant mentions them.

## 10. Stopping and escalation rules

Pause new invitations immediately if a confirmed severe factual error, privacy-boundary breach, credential exposure, or unsafe career recommendation is observed.

The operator should:

1. avoid retaining the sensitive content;
2. record only the anonymous problem category;
3. reproduce the issue using fictional or synthetic input where possible;
4. decide whether the Pilot can safely resume;
5. keep product implementation outside Epic-024.3 unless separately approved.

Ordinary confusion or low usefulness does not stop the Pilot; it is expected discovery evidence.

## 11. Pilot Review Report

After the review gate, create the Epic-024.5 Pilot Review Report with:

### Participation and usage

- eligible primary-audience attempt count;
- broad user-type mix;
- Agent usage paths;
- completion and independent-completion rates;
- common failure reasons.

### Value signals

- report-understanding rate;
- helpfulness rate;
- reuse-intent rate;
- most valuable capabilities;
- common actionable recommendations.

### Quality and technical findings

- confirmed factual errors;
- skill misinterpretations;
- generic or infeasible advice;
- Agent invocation and UX problems;
- reproducible bugs using only fictional or synthetic data.

### Recommendation

Recommend one of:

- continue the Pilot without product changes;
- make a focused documentation or UX correction;
- prepare a narrowly scoped v0.8.1 quality or bug-fix proposal;
- stop the Pilot pending a safety correction.

The report may recommend a v0.8.1 direction, but Epic-024.3 does not implement that version.

## 12. Frozen product scope

Epic-024.3 must not modify:

- Skill Core;
- Application Layer;
- AI Provider behavior;
- CLI behavior;
- Agent Tool schemas;
- `CareerAnalysisReport` or public API contracts;
- backend, frontend, Web delivery, authentication, or persistence.

Allowed work is limited to Pilot execution documents, recruitment copy, privacy-safe record templates, feedback-template alignment, manual feedback classification, and the final Pilot Review Report.

## 13. Implementation deliverables

The execution preparation phase should produce:

```text
docs/pilot-recruitment.md
docs/pilot-record-template.md
docs/pilot-execution-guide.md
```

It should also update `docs/pilot-feedback-template.md` with independent-completion and reuse-intent fields and update project records to mark the Pilot as ready for invitations, not already validated.

Actual participant records and the Pilot Review Report must not be fabricated. They are created only from real, voluntary, privacy-reviewed Pilot attempts.

## 14. Acceptance criteria

Epic-024.3 execution is complete only when:

- 5-10 eligible primary-audience attempts have been conducted, unless a documented severe-safety stop ends the round early;
- the feedback template was usable and collected the required metric fields;
- no prohibited sensitive content was retained in Pilot records;
- activation, value, and quality observations have numerators and denominators;
- output value has initial real-user evidence;
- the Pilot Review Report is generated from real records;
- a focused v0.8.1 direction or continue/stop decision is documented.

Creating preparation documents alone means `Pilot Ready`, not `Pilot Complete`.
