# Epic-020.1 Product Application Layer Boundary Design

## Goal

Add a standalone Product Application Layer that receives a career-analysis
request, orchestrates existing public Skill capabilities, and returns one
structured response. The layer must not change the Skill Core, `skill.ai`, the
Agent Runtime, or the v0.7.0 Skill API.

## Architecture

```text
application/
  schemas/
    career_analysis.py       CareerAnalysisReport and ApplicationResponse
  services/
    career_analysis.py       CareerAnalysisService request entry point
  workflows/
    career_analysis.py       CareerAnalysisWorkflow fail-fast orchestration

CareerAnalysisService
        |
        v
CareerAnalysisWorkflow
        |
        v
NextMoveSkill.run(capability, payload)
        |
        v
Skill Core
```

`application` imports only the stable public Skill interface, Skill API
schemas, and the existing serialization utility. It never imports the Core's
analyzer, matcher, advisor, parser, or other internal capability classes.

`skill` has no import path to `application`; a static dependency test enforces
this one-way direction.

## Workflow

`CareerAnalysisWorkflow.run(resume, job_description)` calls
`NextMoveSkill.run()` in this fixed sequence:

1. `analyze_resume` with `{"resume": resume}`.
2. `improve_resume` with `{"resume": resume}`.
3. `match_job` with `{"resume": resume, "job_description": job_description}`.
4. `career_advice` with `{"resume": resume, "analysis": analysis.result}`.

Each response is inspected before the next step begins. On four successful
responses, the workflow aggregates only their structured results into
`CareerAnalysisReport(resume_analysis, improvement, job_match, career_advice)`
and returns it through `ApplicationResponse(success=True, result=report)`.

## Failure Handling

The workflow is fail-fast. If any `SkillResponse.success` is false, it stops
immediately and returns:

```python
ApplicationResponse(
    success=False,
    error_code="WORKFLOW_STEP_FAILED",
    failed_step=<capability name>,
    message="career analysis workflow failed",
    error=<original SkillResponse.error>,
    result=None,
)
```

The original structured `SkillError` is retained unchanged. The Application
Layer does not continue, compensate, add a fallback, or expose an unhandled
exception. No report is generated on failure.

## Serialization and Data Boundary

`CareerAnalysisReport` and `ApplicationResponse` provide `to_dict()` using
the existing recursive serialization utility. Thus
`json.dumps(response.to_dict())` is supported for both success and failure.

The report carries capability results only. It does not contain the original
resume text, job-description input, provider metadata, prompts, credentials,
or raw provider responses.

## Service Boundary

`CareerAnalysisService` is a thin request entry point. It accepts a
`CareerAnalysisWorkflow` through construction (or builds one from a supplied
`NextMoveSkill`) and delegates `analyze(resume, job_description)` to the
workflow. It contains no HTTP, FastAPI, database, authentication, web, MCP,
agent-adapter, or AI-provider concern.

## Testing

Tests use a mockable `NextMoveSkill.run()` dependency and verify:

- the full four-step capability order and successful report aggregation;
- first-step failure prevents all later calls;
- `match_job` failure prevents `career_advice`;
- all workflow calls use the public Skill API; and
- success and failure responses serialize with `json.dumps(response.to_dict())`.

A dependency-boundary test scans `skill/**/*.py` and confirms that it has no
`application` imports. Existing Core, AI, Agent Runtime, and v0.7.0 API tests
remain part of the full regression suite.

## Scope

This Epic adds no FastAPI endpoint, web UI, database, authentication, user
session, AI provider, MCP integration, or Agent Adapter change. It adds no
fallback behavior and does not modify anything under `skill/`.
