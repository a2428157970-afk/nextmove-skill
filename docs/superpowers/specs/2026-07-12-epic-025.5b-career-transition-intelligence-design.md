# Epic-025.5B Career Transition Intelligence Design

## Decision and scope

Epic-025.5B adds an internal, deterministic Career Transition Intelligence
model. It explains the path from a candidate's current career position to a
target role by classifying the transition, separating direct and transferable
evidence, calibrating transition risk, and producing gap-linked actions.

The new capability answers a different question from matching:

~~~text
Match:      How well does the supplied evidence match the target role now?
Transition: How can the candidate move from the current position to the target?
~~~

This phase is design-only. It creates this specification and its implementation
plan. It must not change `JobMatchResult`, `NextMoveSkill.run()`, the Skill or
Agent schema, Agent Runtime, Application Layer, CLI, providers, dependencies,
databases, network behavior, or Git tags.

## Current Career Intelligence chain

The current internal chain is:

~~~text
ResumeProfile + target job description
  -> DomainClassifier
  -> CareerStageAssessor
  -> MatchScorer requirement evidence
  -> TransferableSkillAssessor
  -> MatchExplanationBuilder
  -> unchanged six-field JobMatchResult
~~~

It can answer:

- which current and target domains/job families are evidenced;
- whether the candidate's stage is entry, developing, experienced, advanced,
  or unknown;
- which target requirements are matched, partial, missing, or unknown;
- which direct or transferable evidence supports the match;
- which strengths, gaps, and risks explain the result.

It cannot yet answer coherently:

- what the candidate's current career position means relative to the target;
- whether the move is same-role, same-domain progression, adjacent-role, or
  cross-domain;
- how difficult the move is and why;
- which gap should be addressed first;
- which concrete action would create the direct evidence required for that
  gap.

`CareerAdvisor` currently emits broad stage- and keyword-based advice. It does
not consume target requirement evidence or transfer limitations, so it is not
the correct owner for evidence-grounded transition planning.

## Alternatives considered

### Add transition fields to JobMatchResult

This would make transition output immediately visible to existing consumers,
but it would change the frozen six-field contract and force changes through the
Skill API, Agent schema, Application Layer, and CLI. It is outside scope and is
rejected.

### Extend CareerAdvisor with target-aware rules

This would reuse an existing career module, but it would mix public generic
career advice with an internal evidence-grounded target transition assessment.
It would also make the advisor depend on matching internals. This is rejected.

### Add an independent internal CareerTransitionAssessor (selected)

Create `skill/career/transition.py` with immutable internal contracts and a
stateless assessor. It consumes existing internal assessments without changing
their public projections. This keeps matching, career stage, transition risk,
and action planning independently testable.

## Architecture and data flow

~~~text
ResumeProfile (transient)                 Target JD (transient)
        |                                        |
        +---------- DomainClassifier ------------+
        |             |                          |
        |       current classification     target classification
        |                                        |
 CareerStageAssessor                             |
        |                                        |
 CareerStageAssessment                           |
        |                                        |
        +------ MatchScorer / JobMatcher internal path
                         |
            MatchAssessment + MatchExplanationResult
                         |
             TransferableSkillAssessment
                         |
              CareerTransitionAssessor
                         |
              CareerTransitionAssessment
                  (internal only)
~~~

The transition assessor consumes structured results. It does not retain raw
resume text or job-description text. Short evidence snippets are the already
filtered `EvidenceItem` values accepted by the matching evidence layer.

## Internal contracts

All contracts live in `skill/career/transition.py`. They are frozen and slotted
dataclasses or string enums and are not exported from the public Skill package.

### Transition type

~~~python
class TransitionType(str, Enum):
    SAME_ROLE = "same_role"
    SAME_DOMAIN = "same_domain"
    ADJACENT_ROLE = "adjacent_role"
    CROSS_DOMAIN = "cross_domain"
    UNKNOWN = "unknown"
~~~

Semantics:

| Type | Rule | Example |
|---|---|---|
| `same_role` | Same known domain and family, with no upward role-level signal | HR Specialist to HR Specialist |
| `same_domain` | Same known domain with a different family or a verified upward role-level signal | HR Specialist to HR Manager |
| `adjacent_role` | A specifically approved directional family pair | Administrative Assistant to HR Assistant |
| `cross_domain` | Different known domains and not an approved adjacent pair | Sales to Product Manager |
| `unknown` | Current or target domain is unknown/other, or evidence is insufficient to choose safely | Uninformative profile or target |

`same_role` is not inferred from family equality alone. A target management or
leadership signal makes HR Specialist to HR Manager `same_domain`, even when
both roles classify to `hr_generalist` in the current small taxonomy.

### Role-level input

The assessor needs a bounded signal to distinguish same-role movement from an
upgrade inside one family:

~~~python
class TargetRoleLevel(str, Enum):
    PEER = "peer"
    MANAGEMENT = "management"
    UNKNOWN = "unknown"
~~~

The caller derives this transient value from deterministic target title and
requirement signals. It is not stored as raw JD text. Recognized management
signals include manager, head, lead, people leadership, team management, and
their maintained Chinese aliases. Ambiguous signals remain `unknown`.

### Capability gap

~~~python
@dataclass(frozen=True, slots=True)
class TransitionCapabilityGap:
    capability: str
    evidence_status: RequirementStatus
    reason: str
    core: bool
~~~

The required field `missing_capabilities` is a tuple of these structured gaps,
not a list of asserted absences. `evidence_status` preserves `partial`,
`missing`, or `unknown`:

- `missing` means supplied evidence explicitly conflicts with the requirement;
- `unknown` means the resume does not provide enough evidence;
- `partial` means transfer or adjacent evidence exists but direct target-role
  ownership remains unproven.

This prevents `unknown` from becoming `missing` inside transition output.

### Risk model

~~~python
class TransitionRiskLevel(str, Enum):
    LOW = "low"
    MEDIUM = "medium"
    HIGH = "high"
    UNKNOWN = "unknown"


@dataclass(frozen=True, slots=True)
class TransitionRiskAssessment:
    level: TransitionRiskLevel
    factors: tuple[str, ...]
    evidence_gaps: tuple[str, ...]
~~~

Risk factors describe evidence conditions, never candidate worth or ability.
Allowed phrasing includes “Product delivery ownership is not directly
evidenced” and “leadership scope needs verification.” Forbidden phrasing
includes “不适合”, “无法转型”, “unsuitable”, and “cannot transition.”

### Recommended action

~~~python
@dataclass(frozen=True, slots=True)
class TransitionAction:
    gap: str
    objective: str
    steps: tuple[str, ...]
    expected_evidence: str
    priority: int
~~~

Every action must reference exactly one capability in `missing_capabilities`.
Actions without a gap are invalid. `expected_evidence` states what truthful,
reviewable direct evidence the action is intended to create; it does not claim
the candidate already has that evidence.

### Assessment result

~~~python
class TransitionConfidence(str, Enum):
    HIGH = "high"
    MEDIUM = "medium"
    LOW = "low"


@dataclass(frozen=True, slots=True)
class CareerTransitionAssessment:
    current_domain: CareerDomain
    target_domain: CareerDomain
    transition_type: TransitionType
    transferable_skills: tuple[str, ...]
    direct_evidence: tuple[EvidenceItem, ...]
    transferable_evidence: tuple[EvidenceItem, ...]
    missing_capabilities: tuple[TransitionCapabilityGap, ...]
    transition_risk: TransitionRiskAssessment
    recommended_actions: tuple[TransitionAction, ...]
    confidence: TransitionConfidence
~~~

The result contains only internal enums, canonical capability names, filtered
evidence items, bounded reasons, and future actions. It contains no profile,
resume body, JD body, identity, employer, credential, provider, endpoint, or
network metadata.

## Assessor interface

~~~python
class CareerTransitionAssessor:
    def assess(
        self,
        current: DomainClassification,
        target: DomainClassification,
        career_stage: CareerStageAssessment,
        transferability: TransferableSkillAssessment | None,
        explanation: MatchExplanationResult,
        target_role_level: TargetRoleLevel = TargetRoleLevel.UNKNOWN,
    ) -> CareerTransitionAssessment:
        ...
~~~

Dependencies are one-way:

- career stage supplies current seniority and confidence signals;
- transferability supplies direct/transfer evidence and explicit limitations;
- explanation supplies requirement states and evidence-grounded gaps;
- the transition assessor aggregates them but does not mutate or replace them;
- `JobMatchResult` remains unchanged and does not contain the assessment.

## Transition classification rules

Rules are evaluated in this order:

1. If current or target domain is `unknown` or `other`, return `unknown`.
2. If current and target domain and family are equal:
   - return `same_domain` when `target_role_level == management` and current
     evidence does not already establish the target leadership scope;
   - otherwise return `same_role`.
3. If domains are equal but families differ, return `same_domain`.
4. If the directional pair is explicitly approved in
   `ADJACENT_TRANSITION_FAMILIES`, return `adjacent_role`.
5. Otherwise, for two different known domains, return `cross_domain`.

The first approved adjacent pair is directional:

~~~python
(JobFamily.ADMINISTRATION, JobFamily.HR_GENERALIST)
~~~

No reverse or transitive relationship is inferred. Sales to Product Manager is
not adjacent merely because transferable evidence exists; it remains
`cross_domain`.

## Direct, transferable, and gap derivation

### Direct evidence

Use `TransferableSkillAssessment.direct_evidence` and matched requirement
evidence. Deduplicate by normalized text and source while preserving order.
Direct evidence supports current readiness but never removes an unrelated gap.

### Transferable skills and evidence

Use only approved `TransferableCapability` entries. The canonical
`target_capability` values become `transferable_skills`; their evidence becomes
`transferable_evidence`. The associated limitation remains available as a risk
factor and gap reason.

### Missing capabilities

Create structured transition gaps from:

- `unknown` target requirements: evidence is insufficient;
- `missing` target requirements: supplied evidence explicitly conflicts;
- `partial` target requirements: transferable or adjacent support exists, but
  direct target ownership is not established.

For action planning, a partial Product delivery mapping becomes a direct
evidence gap such as `Product delivery ownership`, even though matching still
correctly reports the underlying requirement as `partial`. This is not a change
to Match Explanation or `JobMatchResult`.

Core capabilities are defined by the target family taxonomy and maintained
transition action catalog. Product Manager core gaps include Product Planning,
PRD/Requirement Management, and Product Delivery ownership. HR Manager core
gaps include people leadership when management is the target level.

## Transition risk rules

Risk is deterministic and non-compensating. Strong evidence for one capability
cannot hide a different core gap.

### Unknown

Return `unknown` when `transition_type == unknown` or when there are no usable
direct/transfer evidence items and no reliable target requirements. The action
is to provide more current-role and target-role information, not to label the
transition high risk.

### Low

Return `low` when all are true:

- transition is `same_role` or `same_domain`;
- direct matched evidence covers at least 70% of extracted target requirements;
- there is at most one non-core gap;
- no core ownership or leadership gap remains.

### High

Return `high` when either is true:

- transition is `cross_domain`, direct target evidence is sparse, and two or
  more core direct-evidence gaps remain; or
- no usable direct or transferable evidence exists and multiple reliable core
  target requirements remain unsupported.

### Medium

Return `medium` for bounded transitions that do not meet low or high:

- transferable evidence is the main bridge;
- an adjacent-role move has supported overlap but professional target gaps;
- a same-domain upgrade has a leadership or scope gap;
- some direct evidence exists but one or more core capabilities need proof.

Expected calibration:

| Scenario | Expected risk | Reason |
|---|---|---|
| Strong HR Specialist to same HR Specialist role | low | Direct coverage, no core transition gap |
| HR Specialist to HR Manager | medium | Same-domain evidence plus leadership gap |
| Administrative Assistant to HR Assistant | medium | Adjacent support plus recruitment/labor-relations gaps |
| Sales to Product Manager | high | Cross-domain, transfer-led, Product planning/PRD/delivery ownership gaps |

## Action recommendation design

Actions are generated from a small internal catalog keyed by canonical gap.
The assessor never generates free-form generic advice disconnected from a gap.

Example for Product Planning:

~~~text
Gap: Product Planning
Objective: Create reviewable Product Planning direct evidence.
Steps:
  1. Select one fictional or non-confidential user problem.
  2. Document user/problem analysis and success metrics.
  3. Write a bounded PRD and prioritised roadmap.
  4. Produce a prototype or feature design.
  5. Define a data-validation or A/B-test plan.
Expected evidence: A case study containing problem analysis, PRD, roadmap,
prototype, and validation results.
~~~

Other initial catalog entries:

| Gap | Action objective | Expected evidence |
|---|---|---|
| PRD / Requirement Management | Translate a validated problem into testable requirements | Versioned PRD with user stories, acceptance criteria, and decision log |
| Product Delivery ownership | Demonstrate end-to-end delivery in a bounded project | Delivery plan, collaboration log, release result, and retrospective |
| Recruitment | Build supervised or simulated recruitment-process evidence | Fictional role intake, screening rubric, interview workflow, and outcome review |
| Labor Relations | Learn and apply a bounded policy scenario without claiming professional practice | Scenario analysis referencing applicable policy and escalation boundaries |
| People Leadership | Lead a scoped initiative and document team outcomes | Goal, delegation, feedback, decision, and measurable outcome record |

Actions must be safe and feasible:

- use fictional, open, or non-confidential cases;
- never advise fabricating employment experience or metrics;
- distinguish learning artifacts from production ownership;
- avoid legal conclusions for labor-relations scenarios;
- order actions by core gap first, then direct-evidence gap, then optional
  enhancement.

If no catalog entry exists, use a bounded fallback tied to the exact gap:

~~~text
Create one truthful, reviewable work sample for <capability> that documents the
problem, decisions, execution, and outcome. Label simulated work as simulated.
~~~

## Confidence rules

Confidence measures evidence sufficiency for the transition assessment, not the
probability of career success.

- `high`: current and target classification are medium/high confidence, career
  stage is known, target requirements exist, and direct/transfer evidence is
  grounded across at least two relevant capabilities;
- `medium`: domains and requirements are known, but stage or one evidence
  channel is weak;
- `low`: either domain is unknown/other, target requirements are absent, or the
  assessment depends on a single weak signal.

Risk and confidence are independent. A high-risk cross-domain transition may
have high assessment confidence when the evidence and gaps are clear.

## Benchmark expansion

The Epic-025.4 benchmark remains synthetic, offline, content-safe, and
deterministic. Extend its optional scenario expectation with:

~~~python
@dataclass(frozen=True, slots=True)
class ExpectedTransition:
    transition_type: str
    transferable_skills: tuple[str, ...]
    missing_capabilities: tuple[str, ...]
    allowed_risks: tuple[str, ...]
    action_gaps: tuple[str, ...]
    forbidden_conclusions: tuple[str, ...]
~~~

`ScenarioExpectation.transition` is optional so existing non-transition cases
remain compatible. `BenchmarkObservation` gains an internal
`career_transition` value. Reports retain only scenario IDs, check names,
scores, and violations; they do not print resumes, JDs, or evidence text.

### Scenario 1: Sales to Product Manager

- transition type: `cross_domain`;
- transferable: Customer Insight/User Understanding;
- direct-evidence gaps: Product Planning, PRD/Requirement Management, Product
  Delivery ownership;
- risk: `high`, because the path is cross-domain, transfer-led, and retains
  multiple core direct-evidence gaps;
- actions: Product Planning, PRD, and Product Delivery evidence actions;
- forbidden: unsuitable, cannot transition, direct Product experience, owned a
  roadmap, or launched a Product.

### Scenario 2: Administrative Assistant to HR Assistant

- transition type: `adjacent_role`;
- transferable: coordination, documentation, and process support;
- direct-evidence gaps: recruitment and labor relations;
- risk: `medium`;
- actions: recruitment-process evidence and bounded labor-relations learning;
- forbidden: direct recruitment experience, HR expert, or automatic payroll
  and labor-relations competence.

### Scenario 3: HR Specialist to HR Manager

Add a sixth fictional fixture:

- current and target domain: Human Resources;
- transition type: `same_domain`;
- direct strength: HR operational experience;
- core gap: people leadership;
- risk: `medium` (low is allowed only when direct leadership evidence is added
  to a separate strong variant);
- action: lead a scoped HR process initiative and document delegation,
  feedback, decisions, and outcomes;
- forbidden: already managed a team, ready for any management role, or cannot
  become a manager.

### Transition quality metrics

Add internal metrics without changing the public runner contract:

- `transition_type_accuracy`;
- `transition_gap_grounding`;
- `transition_risk_calibration`;
- `transition_action_grounding`;
- `transition_safety_pass_rate`.

Safety gates remain non-compensating:

- unknown is not missing;
- transfer evidence is not direct experience;
- every action maps to a real structured gap;
- no invented current experience;
- no fatalistic conclusion.

## Error and insufficient-information behavior

The assessor returns a structured low-confidence result instead of raising for
ordinary insufficient evidence:

- `transition_type = unknown`;
- `transition_risk.level = unknown`;
- empty direct and transferable evidence;
- no asserted missing capability unless a reliable target requirement exists;
- one information action asking for current role evidence and a specific target
  role.

Programming errors and invalid enum/type construction remain normal Python
errors; no provider fallback or network call is introduced.

## Regression strategy

Implementation must protect:

1. all five existing benchmark scenarios and their 100 scores;
2. the exact six `JobMatchResult` fields and serialized order;
3. `NextMoveSkill.run("match_job", ...)` output compatibility;
4. Agent tools and runtime schemas;
5. `unknown != missing` in both Match and Transition output;
6. direct evidence precedence over transfer evidence;
7. directional, non-transitive Admin-to-HR mapping;
8. no raw resume/JD retention in transition models or reports;
9. no dependency, provider, database, or network change.

Focused tests cover transition classification, risk boundaries, action-gap
linkage, privacy, empty evidence, three benchmark scenarios, and public contract
serialization. Full verification runs:

~~~text
python -m unittest discover -s tests -v
python -m compileall skill tests benchmark
python -m benchmark
git diff --check
~~~

## Acceptance criteria

Epic-025.5B implementation is complete when:

- the internal contracts serialize without public schema changes or raw input;
- all five transition types have deterministic tests;
- Sales-to-Product is cross-domain and transfer-led without claiming Product
  experience;
- Administration-to-HR is adjacent and does not infer recruitment, payroll, or
  labor-relations expertise;
- HR Specialist-to-HR Manager is same-domain and exposes a leadership gap;
- risk factors identify evidence insufficiency without fatalistic language;
- every recommendation references a structured gap and expected direct
  evidence;
- the expanded six-scenario benchmark passes transition and existing safety
  gates;
- all existing tests, compilation, benchmark, and diff checks pass;
- no forbidden boundary, dependency, push, merge, or tag change occurs.
