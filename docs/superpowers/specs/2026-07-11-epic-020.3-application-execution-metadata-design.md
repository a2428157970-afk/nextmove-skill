# Epic-020.3: Application Execution Metadata Contract

## Status and Scope

This specification defines a transport-neutral execution-metadata contract for the Application Layer. It follows Epic-020.1's Application-to-Skill boundary and Epic-020.2's request validation boundary.

The work is intentionally limited to `application/schemas/` and the career-analysis service integration, together with focused tests. It does not add an HTTP API, frontend, database, logging framework, telemetry SDK, provider trace, provider behavior, or third-party dependency.

## Goals

For each valid career-analysis request delegated to the workflow, expose safe, structured metadata that identifies the workflow execution and records its lifecycle:

- a generated execution identity;
- a stable workflow name;
- lifecycle status (`started`, `completed`, or `failed`);
- UTC, timezone-aware start and completion times; and
- the workflow step that failed, when a workflow step fails.

The contract must be immutable, JSON serializable through `to_dict()`, and safe to attach to an `ApplicationResponse`.

## Non-goals and Data Boundary

`ExecutionMetadata` is operational context, not a request, result, trace, or audit record. It must not contain or derive fields for:

- resume content or a resume profile;
- job-description content;
- prompts, credentials, provider configuration, provider responses, or provider traces;
- Skill capability inputs or unredacted outputs; or
- backend, frontend, HTTP, persistence, logging, or telemetry state.

The Skill Core remains unaware of this type. Application code continues to call only the public `NextMoveSkill.run()` interface, and `skill/` must not import `application/`.

## Considered Approaches

### A. Service-owned lifecycle (selected)

`CareerAnalysisService` creates metadata after request validation and replaces it with a terminal immutable value after `CareerAnalysisWorkflow.run()` returns. The workflow continues to return an `ApplicationResponse` describing the business outcome, and the service attaches the final metadata.

This keeps clock access, execution-id generation, and lifecycle ownership at the Application boundary. It avoids changing the public Skill contract or making the workflow responsible for transport-adjacent operational state.

### B. Workflow-owned lifecycle

The workflow could generate and finalize metadata itself. This would make its return value self-contained, but unnecessarily couples orchestration logic to time and identity generation and makes the service unable to define the execution boundary cleanly.

### C. External observer or callback

An observer could receive lifecycle events and later produce metadata. This is not selected because it introduces an additional extension point with no consumer in this Epic and resembles logging or telemetry scope that is explicitly excluded.

## Schema Contract

Add `application/schemas/execution.py` with an immutable data object:

```python
@dataclass(frozen=True, slots=True)
class ExecutionMetadata:
    execution_id: str
    workflow_name: str
    status: str
    started_at: datetime
    completed_at: datetime | None = None
    failed_step: str | None = None

    def to_dict(self) -> dict[str, object]: ...
```

`execution_id` is generated by the Application Layer with the Python standard library `uuid.uuid4()` and exposed as a string. It is opaque: consumers must not infer request, user, provider, or content information from it.

`workflow_name` is the stable literal `"career_analysis"` for this workflow. It identifies the application workflow, never a provider or transport.

`status` is restricted to exactly `"started"`, `"completed"`, and `"failed"`. The schema validates its invariant at construction time. `started` has no completion time or failed step; `completed` has a completion time and no failed step; `failed` has both a completion time and a failed step. A completion time must not precede the start time.

Both time fields use `datetime` internally. `started_at` is required and `completed_at` is optional. Every supplied time must be timezone-aware and in UTC; naive values such as `datetime.now()` are invalid. Service code obtains the current time exclusively with `datetime.now(timezone.utc)`.

`to_dict()` returns only the six contract fields. It serializes time values with `datetime.isoformat()`, producing UTC ISO 8601 strings such as `"2026-07-11T12:00:00+00:00"`; a missing completion time becomes `null`. The returned mapping is directly accepted by `json.dumps()` and never contains numeric timestamps or objects from third-party date libraries.

Because the object is frozen, lifecycle progression creates a new metadata instance (or an equivalent standard-library `dataclasses.replace` result) instead of mutating the started instance.

## Response Contract

Extend `ApplicationResponse` with this optional field, after the existing fields:

```python
metadata: ExecutionMetadata | None = None
```

All current constructor call sites remain valid because the new field has a default value. `ApplicationResponse.to_dict()` retains its existing success and failure shape when `metadata is None`, preserving older serialization callers. When metadata is supplied, `to_dict()` adds a top-level `"metadata"` key whose value is `metadata.to_dict()`.

The response still owns only its established fields (`success`, `result`, `error_code`, `failed_step`, `message`, and `error`) plus the safe metadata object. It must not retain the original `CareerAnalysisRequest`, raw Skill input mappings, or any additional Skill result content.

## Service Lifecycle Integration

`CareerAnalysisService.analyze()` has the following bounded lifecycle:

1. Validate `CareerAnalysisRequest` exactly as it does today. A validation failure occurs before any workflow execution; it remains an `APPLICATION_VALIDATION_ERROR` response with `metadata=None`.
2. For a valid request, generate an opaque `execution_id` and create a `started` `ExecutionMetadata` for `career_analysis` using `datetime.now(timezone.utc)`.
3. Call `CareerAnalysisWorkflow.run()` with the validated, normalized values. No metadata is passed to the workflow or to `NextMoveSkill.run()`.
4. If the workflow response succeeds, create terminal metadata with status `completed`, a UTC completion time, and `failed_step=None`; return an equivalent successful `ApplicationResponse` carrying that metadata.
5. If the workflow response fails, preserve its existing error fields and `failed_step`; create terminal metadata with status `failed`, a UTC completion time, and the same `failed_step`; return the response with that metadata.

This design retains the fixed fail-fast capability order: `analyze_resume`, `improve_resume`, `match_job`, and `career_advice`.

## Dependency Boundary

The permitted data flow is:

```text
CareerAnalysisService
    -> ExecutionMetadata
    -> ApplicationResponse

CareerAnalysisService
    -> CareerAnalysisWorkflow
    -> NextMoveSkill.run()
```

There is no metadata path into the Skill Core, AI Provider, backend, or frontend. `application` may keep importing only public Skill interfaces (`skill`, `skill.schemas`, and `skill.utils` as already enforced). The new schema uses only standard-library modules such as `dataclasses`, `datetime`, and `uuid`.

## Test Plan

Add `tests/test_application_execution.py` with focused unit tests for:

1. a successful service execution: non-empty execution ID, workflow name, `completed` status, UTC-aware start and completion times, and no failed step;
2. a failed workflow execution: `failed` status, completion time, and metadata `failed_step` equal to the existing response `failed_step`;
3. preservation of the first fail-fast workflow failure step;
4. `ExecutionMetadata.to_dict()` and `ApplicationResponse.to_dict()` JSON serialization through `json.dumps()` with ISO 8601 strings, not timestamps;
5. construction-time rejection of invalid status, naive/non-UTC timestamps, invalid lifecycle combinations, and completion before start;
6. absence of sensitive or forbidden fields by asserting the exact serialized metadata key set;
7. backward compatibility: an `ApplicationResponse` without metadata keeps its current serialized form; and
8. dependency direction: extend the existing application static-scan coverage to include the new module and ensure no forbidden Skill implementation, provider, backend, or frontend imports are introduced.

The implementation verification gate will run the full test suite, compile `application`, `skill`, and `tests`, and run `git diff --check`. No live provider, transport, or dependency-installation test is required.

## Acceptance Criteria

- Metadata is created only after validation succeeds and is terminal when the service returns from a workflow invocation.
- All metadata times are aware UTC `datetime` values internally and ISO 8601 `+00:00` strings in serialized output.
- Existing response callers work without providing metadata and preserve their old serialized shape when none is present.
- Workflow failures retain the existing structured `SkillError` and failed step while exposing the same failed step in metadata.
- The contract is immutable, JSON serializable, content-blind, and free of transport, provider, persistence, logging, telemetry, and new dependencies.
