# Epic-025.3 Evidence-based Match Explanation Design

## Decision and scope

Epic-025.3 freezes an offline, deterministic explanation layer for job matching. The internal flow becomes:

~~~text
ResumeProfile + job description
  -> domain-aware requirement extraction and scoring
  -> requirement evidence mapping
  -> strengths, gaps, and risks
  -> MatchExplanationResult (internal)
  -> unchanged JobMatchResult (public)
~~~

This Epic is design-only. It must not change `NextMoveSkill.run()`, public method signatures, `JobMatchResult`, Application Layer code, Agent schemas/contracts, CLI, providers, databases, dependencies, frontend, backend, release files, or tags. The dependency direction remains `application -> public Skill`.

## Current architecture and baseline

The checked-out matching path is:

- `skill/matching/matcher.py` owns keyword extraction, scoring, and public string-list explanations in `JobMatcher`.
- `skill/matching/schemas.py` owns `JobMatchResult`; `skill/schemas/matching.py` re-exports it.
- `skill/interface.py` calls `JobMatcher.match(profile, job_description)` from `match_job()` and the fixed-order `career_analysis` workflow.
- `skill/utils.to_dict()` serializes every dataclass field recursively.
- `skill/agent/tools.py` fixes the Agent-visible `JOB_MATCH_OUTPUT_SCHEMA` to the same six keys.

The public result is exactly:

~~~python
JobMatchResult(
    match_score: int,
    matched_skills: list[str],
    missing_skills: list[str],
    strengths: list[str],
    gaps: list[str],
    recommendations: list[str],
)
~~~

Today the matcher extracts a fixed English technical-keyword list. It scores matched keywords at 80%, adds 20% for any experience entry, and has no per-requirement evidence object or source retention. It cannot distinguish an absent capability from a capability not discussed in the resume. `tests/test_job_matcher.py` only covers technical-keyword cases; serialization, Agent, interface, workflow, API, and acceptance tests form the public compatibility boundary.

### Baseline reconciliation finding

The repository contains committed Epic-025.1 and Epic-025.2 design/plan documents, but the current `skill/matching/` directory does not contain the planned `taxonomy.py`, `classifier.py`, or `scoring.py` modules. It still contains the original global-keyword matcher. This design records that difference; it does not implement or substitute for Epic-025.1.

Epic-025.3 implementation has a hard precondition: the actual Epic-025.1 domain-aware implementation, including its internal `MatchAssessment`, must be checked out and passing. If absent, work stops and requests the correct commit/branch rather than silently rebuilding prior-Epic behavior.

## Alternatives

### A. Add prose directly in JobMatcher

Smallest change, but scoring, evidence, and language remain coupled. It makes deterministic testing and the `unknown`/ `missing` safety rule unreliable. Rejected.

### B. Internal requirement assessments and explanation model (selected)

Extend the internal domain-aware scorer with immutable requirement assessments, then derive `MatchExplanationResult`. `JobMatcher` maps it to the six legacy fields. This keeps one scoring source of truth and preserves callers.

### C. Public explanation field

Rich evidence would be directly consumable but would change serialized `JobMatchResult` and the Agent contract. Explicitly deferred to a separately approved versioned contract.

## Requirement Evidence Mapping

### Internal types

Future implementation adds internal frozen, slotted types under `skill/matching/schemas.py`. They are not re-exported through `skill.schemas`, attached to public results, or added to Agent schemas.

~~~python
class RequirementStatus(str, Enum):
    MATCHED = "matched"
    PARTIAL = "partial"
    MISSING = "missing"
    UNKNOWN = "unknown"

class EvidenceConfidence(str, Enum):
    HIGH = "high"
    MEDIUM = "medium"
    LOW = "low"

@dataclass(frozen=True, slots=True)
class EvidenceItem:
    text: str
    source: str  # summary, experience, project, skill, education, certification, language

@dataclass(frozen=True, slots=True)
class RequirementEvidence:
    requirement: str
    kind: str  # skill, responsibility, qualification, preferred
    status: RequirementStatus
    evidence: tuple[EvidenceItem, ...]
    confidence: EvidenceConfidence

@dataclass(frozen=True, slots=True)
class MatchExplanationResult:
    requirements: tuple[RequirementEvidence, ...]
    strengths: tuple[MatchStrength, ...]
    gaps: tuple[MatchGap, ...]
    risks: tuple[MatchRisk, ...]
~~~

Canonical requirement names come from the Epic-025.1 requirement extractor. Evidence retains short supplied phrases and professional sources only. It is ordered by job requirement, then resume source order: summary, experience, projects, skills, education, certifications, languages. Equivalent normalized text/source pairs are deduplicated. Contact information is excluded.

Example:

~~~json
{
  "requirement": "鎷涜仒宸ヤ綔",
  "kind": "responsibility",
  "status": "matched",
  "evidence": [

    {"text": "3涓湀瀹屾垚1450浜虹绾?, "source": "experience"}

  "confidence": "high"
~~~

### Status rules

| Status | Meaning | Evidence rule | Required wording |
|---|---|---|---|
| `matched` | Full direct support | Direct alias/responsibility evidence; high confidence needs two independent items or explicit scoped/quantified evidence | State support with evidence. |
| `partial` | Some support only | Adjacent or component-level evidence, without proving the remaining component | State both supported and unproven portions. |
| `missing` | Affirmatively disproved or conflicted | Explicit negative statement or explicit qualification/credential/eligibility conflict | State the factually supported conflict. |
| `unknown` | Insufficient evidence | No direct, adjacent, or negative evidence | Say 鈥渘ot evidenced in the resume鈥? never 鈥渓acks鈥? |

Non-mention is always `unknown`, not `missing`. A skills list that does not include a JD term is not proof that the person lacks it. `missing` is intentionally rare. Confidence measures sufficiency, not candidate quality: high for direct/scoped evidence, medium for one direct item, low for sparse, adjacent, ambiguous, or unknown evidence.

## Strength model

Internal `MatchStrength` contains category, summary, evidence, and confidence. It is emitted only for `matched` or directly transferable `partial` requirements. It cannot infer scope, seniority, metrics, or delivery experience.

Categories:

- `requirement_coverage`: direct required skill, responsibility, or qualification.
- `scope_or_impact`: explicit scale, quantified outcome, process ownership, or system complexity attached to a matching requirement.
- `domain_depth`: multiple direct requirements in the selected job family, supported by independent evidence.
- `transferable_strength`: adjacent-domain capability that supports a target requirement without proving target-domain delivery.

Static, bounded taxonomy cues may recognize HR recruitment scale, attendance scale, payroll support, and employee relations; or technology stack coverage, architecture, performance, reliability, and deployment complexity. Sales customer discovery and stakeholder coordination can be transferable for product roles, but cannot become product-delivery evidence.

## Gap and risk model

Internal `MatchGap` has requirement, kind, status, summary, and supporting evidence. Internal `MatchRisk` has category, summary, related requirements, and confidence.

Gap categories:

- `missing_capability`: `missing` only; explicit negative/conflicting evidence required.
- `insufficient_evidence`: `unknown`; asks for truthful evidence or clarification.
- `partial_coverage`: `partial`; names the unproven component.
- `qualification_conflict`: explicit degree, credential, eligibility, or language conflict.

Risk categories:

- `evidence_risk`: a material requirement is unknown or partial.
- `transition_risk`: different non-adjacent families plus material target responsibilities without direct delivery evidence.
- `scope_risk`: job requires explicit scale, ownership, or lifecycle responsibility but supplied evidence is narrower/indirect.
- `qualification_risk`: explicit qualification mismatch.

## Output Contract Strategy

`MatchExplanationResult` exists only for one matching call: the scorer/explanation builder creates it, then `JobMatcher` immediately maps it. It has no public getter, cache, public result field, Agent key, CLI key, or Application dependency.

| Legacy field | Internal source | Safety rule |
|---|---|---|
| `match_score` | Epic-025.1 `MatchAssessment` score | Explanation never independently changes weights or score. |
| `matched_skills` | Canonical skill requirements marked `matched` | Skills only; deterministic order. |
| `missing_skills` | Canonical skill requirements marked `missing` | Never include unknowns, partial responsibilities, or prose. |
| `strengths` | High-value strength summaries | Cite supplied evidence only. |
| `gaps` | Gap and risk summaries | Unknown means 鈥渘ot evidenced鈥? never 鈥渕issing鈥? |
| `recommendations` | Gap/risk actions | Request evidence, clarification, or truthful development; never invent claims. |

This preserves the exact six fields, types, defaults, serialized key order, Agent schema, `match_job` signature, and fixed `career_analysis` workflow.

## Determinism and error cases

Blank/uninformative JDs keep the existing zero score, produce no invented missing skills, and yield one low-confidence request for JD detail. Sparse resumes make requirements `unknown` unless a qualification conflict is explicit. Ties and ambiguous aliases remain unknown/low confidence. Processing is local, deterministic, Unicode-normalized, multilingual-ready, dependency-free, and does not log or persist resume/JD content.

## Testing strategy

TDD fixtures must be synthetic and cover:

1. **HR specialist -> HR administrative role:** recruitment, attendance, payroll support, and employee relations are matched; unmentioned employee-file/administrative work is unknown, not missing; scale can support a strength.
2. **Backend engineer -> Python backend role:** stack and project-complexity evidence create strengths; unmentioned explicit requirements are unknown absent explicit negative evidence.
3. **Sales manager -> product manager:** customer-needs analysis is transferable/partial; product planning, engineering coordination, delivery, and lifecycle are unknown where not evidenced; a truthful transition risk is emitted.
4. **Insufficient information:** no fabricated skill/evidence, stable low-confidence result, and unknown remains distinct from missing.

Each fixture verifies canonical source text, stable ordering, no contact-data use, no fabricated metrics, and truthful recommendations. Regressions assert `JobMatchResult` fields and serialized keys, Agent schema, `NextMoveSkill.run()`, CLI, and application import boundaries remain unchanged.

## Future implementation acceptance

- Every public explanation string maps to requirement evidence.
- Non-mention never becomes missing.
- Score remains owned by Epic-025.1 assessment.
- All four fixture classes pass.
- No prohibited boundary, provider, dependency, database, CLI, Application, or tag changes occur.
