# Epic-020.2 Application Request Boundary Validation Design

## Goal

Add an application-owned request contract and validation boundary for career
analysis. Invalid input must return a structured application response before
the workflow or Skill is invoked. The fixed four-step workflow and the Skill
Core remain unchanged.

## Scope and Constraints

This Epic changes only the application request boundary, application response
contract, its service entry point, and their tests. It adds no HTTP endpoint,
FastAPI, frontend, database, authentication, provider behavior, credential
handling, prompt data, or dependency.

The dependency direction remains:

```text
application -> public skill API -> Skill Core
```

`skill/` must not import `application/`.

## Request Contract

`application/schemas/request.py` defines `CareerAnalysisRequest` with:

- `resume: ResumeProfile | str`
- `job_description: str | None = None`

The request accepts the same two resume representations that the public
`NextMoveSkill` accepts. A string resume must contain at least one
non-whitespace character. A `ResumeProfile` is accepted without inspecting or
mutating its content. Any other resume type is rejected.

`job_description` may be a string or `None`; any other type is rejected. It
is optional: `None` is normalized to `""` after validation. The request object
does not define, retain, or serialize credentials, provider data, prompts, or
raw provider responses.

The request exposes validation with explicit messages:

- `resume must be a ResumeProfile or str`
- `resume must not be empty`
- `job_description must be a str or None`

## Service Flow

`CareerAnalysisService.analyze(request)` is the application entry point.

```text
CareerAnalysisRequest
        |
        v
validate and normalize optional job description
        |
        +-- invalid --> ApplicationResponse(APPLICATION_VALIDATION_ERROR)
        |
        v
CareerAnalysisWorkflow.run(resume, normalized_job_description)
        |
        v
NextMoveSkill.run() fixed four-step sequence
```

For validation failure, the service returns an `ApplicationResponse` with:

- `success=False`
- `result=None`
- `error_code="APPLICATION_VALIDATION_ERROR"`
- `failed_step="request_validation"`
- an explicit validation `message`
- `error=SkillError(code="APPLICATION_VALIDATION_ERROR", message=<same message>)`

The service must not call the workflow or Skill in this case. It does not
catch, translate, or otherwise alter workflow failures.

For valid requests with `job_description=None`, the service supplies `""` to
the unchanged workflow. The workflow still calls `analyze_resume`,
`improve_resume`, `match_job`, and `career_advice` in that order. Application
does not infer, generate, or recommend a job description.

## Response Contract

`ApplicationResponse` continues to represent two JSON-serializable outcomes:

```python
{"success": True, "result": CareerAnalysisReport}
```

and

```python
{
    "success": False,
    "result": None,
    "error_code": str,
    "failed_step": str,
    "message": str,
    "error": SkillError,
}
```

`to_dict()` recursively serializes the report and error through the existing
Skill serialization utility, so `json.dumps(response.to_dict())` works for
success, workflow failure, and application validation failure. Successful
responses contain no error data; failure responses contain no report.

## Tests

Create `tests/test_application_request.py` to cover:

1. valid string request;
2. empty or whitespace-only resume rejection;
3. omitted `job_description` normalization to `""` while all four workflow
   calls execute;
4. invalid resume and job-description types;
5. validation failure does not call workflow or Skill;
6. successful workflow response is returned unchanged;
7. workflow failure response is returned unchanged; and
8. JSON serialization for validation, success, and workflow failures.

Keep the existing static dependency test that rejects `application` imports
under `skill/`. Run the complete unit suite, compile all three packages, and
run `git diff --check` before completion.
