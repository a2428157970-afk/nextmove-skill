# Epic-025.4 Career Intelligence Quality Benchmark Design

## Decision and scope

Epic-025.4 defines an offline, deterministic quality benchmark for the combined
career-intelligence path:

~~~text
synthetic ResumeProfile + synthetic job description
  -> DomainClassifier
  -> CareerStageAssessor
  -> MatchScorer requirement evidence
  -> MatchExplanationBuilder
  -> unchanged JobMatchResult
  -> benchmark metrics + content-safe review summary
~~~

The selected design is a hybrid benchmark: machine-checkable scenario
expectations and hard safety gates are combined with a five-item human review
checklist. Exact prose snapshots are deliberately excluded because harmless
wording changes must not look like quality regressions.

This design phase changes no business code. It must not modify
`NextMoveSkill.run()`, `JobMatchResult`, the Skill API, Agent contract,
Application Layer, CLI, providers, dependencies, release files, or Git tags.
Only this specification and its implementation plan are in scope.

## Current quality capability analysis

### `skill/matching/`

- `taxonomy.py` defines a focused v1 domain and job-family taxonomy.
- `classifier.py` performs deterministic resume and JD domain classification.
- `scoring.py` owns domain, requirement, and qualification scoring and maps
  professional evidence to `matched`, `partial`, `missing`, or `unknown`.
- `explanations.py` derives internal strengths, gaps, and transition risks from
  requirement evidence without changing the score.
- `matcher.py` composes those capabilities and maps the internal explanation to
  the unchanged six-field `JobMatchResult`.

### `skill/career/`

- `stage_assessor.py` derives `entry`, `developing`, `experienced`, `advanced`,
  or `unknown` from experience, responsibility, and impact signals.
- `advisor.py` consumes the internal assessment while preserving the legacy
  public career-level contract and generates bounded actions.

### `skill/analysis/`

- `analyzer.py` checks resume completeness, visible skills, projects,
  experience, and quantified outcomes.
- It delegates career-level inference to `CareerStageAssessor`, so analysis and
  career advice share one stage model.

### Existing quality evidence

The repository has focused unit and facade tests for classification, scoring,
career stage, evidence mapping, explanation building, public serialization,
Agent schemas, and workflow compatibility. These tests prove individual rules,
but they do not provide one versioned scenario dataset, cross-capability metric
summary, regression runner, or human review standard.

### Missing benchmark capabilities

- No canonical scenario benchmark spanning resume analysis, stage, matching,
  evidence, and explanation.
- No explicit quality metrics or release-neutral pass thresholds.
- No reusable synthetic regression samples for the combined pipeline.
- No common human evaluation rubric for meaning, grounding, risk, and action.
- No content-safe report that identifies failures without retaining inputs.

The review also identifies two current v1 boundaries that the benchmark must
expose rather than hide: Product Manager is not a taxonomy family, and
Administration to Human Resources is not an `ADJACENT_DOMAINS` pair. Scenario
expectations represent desired career-intelligence quality; initial semantic
failures become explicit benchmark findings and are not silently rewritten to
match current behavior.

## Alternatives considered

### Exact output snapshots

Store complete public and internal output for each case. This is simple but
brittle: punctuation, ordering, or safer wording changes would create noisy
failures. Rejected.

### Human-only review

Allow reviewers to score free-form examples. This handles nuance but cannot
protect deterministic safety rules or run in regression. Rejected.

### Structured expectations plus human review (selected)

Store only synthetic inputs and atomic expectations: domains, stage,
requirement statuses, required explanation concepts, and forbidden
conclusions. An offline evaluator calculates metrics and safety gates, while a
reviewer scores the semantic result using anchored 1–5 criteria.

## Benchmark architecture

Implementation should remain outside production packages:

~~~text
tests/fixtures/career_intelligence/
  five versioned JSON scenarios
        |
        v
tests/quality/career_intelligence_benchmark.py
  loader + pipeline adapter + metric evaluator
        |
        +--> tests/test_career_intelligence_benchmark.py
        +--> tests/test_career_intelligence_scenarios.py
        +--> scripts/run_career_intelligence_benchmark.py
~~~

The pipeline adapter may call the internal `CareerStageAssessor` and
`JobMatcher._assess_and_explain()` because this is an internal quality harness,
not a public consumer. It must separately execute `JobMatcher.match()` to check
the public six-field result. It must not add a benchmark hook to production
classes.

The runner emits only case identifiers, metric scores, failed check names, and
aggregate counts. It never writes resume text, JD text, evidence snippets,
personal information, employers, credentials, or raw explanation prose.

## Scenario dataset contract

Each UTF-8 JSON fixture uses this conceptual shape:

~~~json
{
  "case_id": "hr-strong-match",
  "description": "Same-domain HR match",
  "profile": {
    "summary": "Synthetic professional summary",
    "skills": ["Recruitment"],
    "experience": [{"role": "Recruitment Specialist", "start_date": "2021-01", "end_date": "2025-01", "highlights": ["Synthetic achievement"]}],
    "education": [],
    "projects": [],
    "certifications": [],
    "languages": []
  },
  "target": {
    "role": "HR Specialist",
    "job_description": "Synthetic requirements"
  },
  "expected": {
    "resume_domain": "human_resources",
    "target_domain": "human_resources",
    "resume_family": "recruitment",
    "target_family": "hr_generalist",
    "career_stage": "developing",
    "requirement_statuses": {"Recruitment": "matched"},
    "strength_concepts": ["Recruitment"],
    "gap_concepts": [],
    "risk_categories": [],
    "forbidden_conclusions": ["cannot perform the role"]
  }
}
~~~

Fixtures contain no `personal_information`, `raw_text`, links, real company,
real person, email, phone, address, endpoint, token, or credential. Evidence is
limited to fictional professional facts needed for the expected decision.
Expectation matching is concept- and category-based, not full-string equality.

## Benchmark scenarios

### Scenario 1: Same-domain strong match — HR to HR Specialist

- **Input profile:** Four years as a Recruitment Specialist; recruitment and
  attendance skills; owned campus recruitment and documented a fictional hiring
  scale. No personal information.
- **Target role:** HR Specialist requiring recruitment, attendance, and
  administrative support.
- **Expected domain:** Resume and target are `human_resources`.
- **Expected career stage:** `developing`; duration and ownership support the
  stage, but the fixture does not claim advanced strategic or people scope.
- **Expected strengths:** Recruitment and attendance; hiring scale may support
  a `scope_or_impact` strength only when tied to supplied evidence.
- **Expected gaps:** Administrative support is `unknown` if not supplied, never
  `missing`.
- **Expected risks:** Evidence verification for the unproven administrative
  responsibility; no domain-transition risk.
- **Forbidden conclusions:** The candidate lacks administrative ability, is an
  advanced HR leader, or has any unsupplied employee-relations experience.

### Scenario 2: Technical role match — Backend Engineer

- **Input profile:** Six years of backend work; Python, FastAPI, SQL, and Docker;
  a complex service migration with a fictional quantified reliability outcome;
  a Bachelor's degree explicitly supplied.
- **Target role:** Backend Engineer requiring Python, FastAPI, API, SQL, Docker,
  and a Bachelor's degree.
- **Expected domain:** Resume and target are `technology`, family `backend`.
- **Expected career stage:** `experienced`; duration, complex cross-functional
  responsibility, and impact are all evidenced.
- **Expected strengths:** Direct technical stack coverage, qualification
  coverage, and only the supplied scale/impact.
- **Expected gaps:** None for the listed required concepts; an unmentioned cloud
  platform, if present in the JD, must be `unknown`.
- **Expected risks:** No transition risk; only an evidence risk for any explicit
  but unmentioned optional requirement.
- **Forbidden conclusions:** Kubernetes, cloud delivery, team leadership, or
  architecture ownership unless the fixture explicitly supplies it.

### Scenario 3: Cross-career transition — Sales to Product Manager

- **Input profile:** Seven years in sales/account management with customer
  discovery, commercial analysis, stakeholder coordination, and a fictional
  quantified outcome; no product-delivery claim.
- **Target role:** Product Manager requiring customer insight, commercial
  understanding, product planning, and product delivery.
- **Expected domain:** Resume is `sales`. The target is `other` under the frozen
  v1 taxonomy because Product Manager has no current family; this boundary must
  be reported explicitly rather than relabelled as technology or marketing.
- **Expected career stage:** `experienced`; career stage is transferable and
  must not be reset to entry solely because the target occupation changes.
- **Expected strengths:** Customer insight and commercial understanding as
  transferable evidence.
- **Expected gaps:** Product planning and delivery are `unknown` when not
  supplied, not explicit missing capabilities.
- **Expected risks:** `domain_transition` and `career_transition`, with product
  delivery identified as needing verification.
- **Forbidden conclusions:** “Unsuitable”, “cannot perform”, “lacks product
  delivery experience”, or invented product ownership, roadmap, launch, or
  engineering-delivery experience.

### Scenario 4: Adjacent-role migration — Administrative Assistant to HR Assistant

- **Input profile:** Three years of office administration, onboarding document
  coordination, attendance record support, and stakeholder coordination.
- **Target role:** HR Assistant requiring attendance, onboarding administration,
  recruitment coordination, and employee-file support.
- **Expected domain:** Resume is `operations` / `administration`; target is
  `human_resources` / `hr_generalist`.
- **Expected career stage:** `developing`.
- **Expected strengths:** Attendance and onboarding administration are
  transferable, partial evidence for the target role.
- **Expected gaps:** Recruitment coordination and employee-file support remain
  `unknown` unless directly supplied.
- **Expected risks:** `partial_coverage` for the adjacent responsibilities and a
  bounded transition/evidence risk. The benchmark deliberately exposes that
  this adjacency is not represented in the current taxonomy.
- **Forbidden conclusions:** Direct recruitment experience, full HR-generalist
  coverage, employee-relations expertise, or a claim that the candidate lacks
  those capabilities.

### Scenario 5: Insufficient information

- **Input profile:** Empty structured profile with no raw text or identity.
- **Target role:** A weak description with no classifiable occupation or
  requirement.
- **Expected domain:** Resume and target are `unknown`.
- **Expected career stage:** `unknown` with low confidence.
- **Expected strengths:** None.
- **Expected gaps:** Only insufficient-information guidance; no named missing
  skill.
- **Expected risks:** None beyond low confidence/insufficient evidence.
- **Forbidden conclusions:** Any domain, stage, skill, qualification, work
  history, metric, employer, suitability decision, or `missing_skills` entry.

## Automated quality metrics

Every metric is an integer percentage from 0 to 100. Expectations are atomic,
so a metric is `100 * passed expectations / applicable expectations`; a metric
with no applicable expectation is recorded as `not_applicable` and excluded
from averages.

### Matching Quality

- **Domain accuracy:** Correct resume domain, target domain, and optional resume
  and target job-family expectations when the fixture defines them.
- **Requirement coverage:** Expected canonical requirements are present exactly
  once and have the expected `matched`, `partial`, `missing`, or `unknown`
  status.

### Evidence Quality

- **Evidence relevance:** Evidence attached to an expected requirement uses an
  allowed professional source and contains one of that expectation's supplied
  evidence concepts.
- **Evidence grounding:** Every retained evidence item can be found verbatim in
  the fixture's allowed professional fields. Contact information and invented
  text always fail this metric.

### Explanation Quality

- **Strength correctness:** Every required strength concept is supported by a
  matched requirement with medium/high confidence, and no forbidden strength is
  emitted.
- **Gap correctness:** Expected unknown/missing concepts use the correct status
  and wording semantics; unknown cannot be phrased as affirmative absence.
- **Risk correctness:** Expected risk categories and related requirements are
  present, with no unsupported transition, scope, or qualification risk.

### Safety Quality

Safety checks are hard gates, not compensating score points:

- **Unknown is not missing:** No `unknown` requirement enters
  `missing_skills` or an affirmative missing-capability conclusion.
- **No invented experience:** Every experience, responsibility, scope, metric,
  credential, and qualification claim is grounded in the fixture.
- **No unsupported claims:** No forbidden conclusion, suitability verdict, raw
  input, identity, credential, provider detail, or network detail appears.

A scenario passes automatically only when all safety gates pass, every
applicable metric is at least 80, and the scenario average is at least 85. The
dataset passes when all five scenarios pass and aggregate domain accuracy is
100. Until a desired semantic expectation is supported by the production
baseline, the runner reports a benchmark failure; unit-test integration must
not weaken or rewrite the expectation to make the baseline green.

## Human evaluation: Career Intelligence Review Checklist

Reviewers score each item from 1 to 5 using the output and the synthetic source
fixture. They must not copy fixture or output prose into the review record.

| Criterion | 1 — unacceptable | 3 — adequate | 5 — excellent |
|---|---|---|---|
| Correctly understands the candidate background | Misclassifies the main background or stage | Main background is correct with minor loss of nuance | Domain, stage, transferable context, and uncertainty are all correct |
| Uses resume evidence | Important conclusions are invented or untraceable | Most conclusions have supplied support | Every material conclusion is traceable and evidence relevance is clear |
| Avoids overstating the match | Makes suitability verdicts or turns unknown into missing | Mostly calibrated, with minor overstatement | Consistently distinguishes direct, partial, missing, and unknown evidence |
| Identifies key risks | Omits or invents material risks | Names the main risk without full precision | Names each material risk, its evidence boundary, and what needs verification |
| Gives actionable advice | Advice is generic, unsafe, or asks for fabrication | At least one relevant next action | Actions are specific, truthful, feasible, and tied to a gap or risk |

Human acceptance requires an average of at least 4.0, no criterion below 3, and
all automated safety gates passing. A review record contains only `case_id`,
five integer scores, reviewer role, review date, optional content-free reason
codes, and pass/fail. It contains no resume, JD, evidence, explanation text,
identity, employer, or free-form private data.

## Regression and reporting strategy

- Loader and evaluator tests prove stable ordering, schema rejection, metric
  math, hard safety gates, grounding, and content-safe serialization.
- Scenario tests execute all five fixtures through the internal pipeline and
  verify atomic expectations independently of exact prose.
- Public regressions reassert the six `JobMatchResult` fields, serialized key
  order, Agent schema, `NextMoveSkill.run("match_job")`, and Application Layer
  dependency direction without modifying those boundaries.
- The standalone runner supports JSON and concise text summaries, returns zero
  only when all benchmark thresholds pass, and reports no source/output text.
- The ordinary unit suite must remain green. Known semantic benchmark failures
  are reported by the runner and become inputs to a separately approved
  remediation Epic; evaluator mechanics and safety gates remain regression
  tests immediately.

## Acceptance criteria

- Exactly five initial synthetic scenarios cover strong same-domain matching,
  technical matching, cross-career transition, adjacent-role migration, and
  insufficient information.
- Each scenario defines the required input, target, expected domains and stage,
  strengths, gaps, risks, and forbidden conclusions.
- All seven semantic metrics and three hard safety gates are deterministic and
  content-safe.
- The five-question human checklist has anchored 1–5 scoring and an explicit
  acceptance rule.
- The benchmark introduces no production hook, provider, network call,
  credential, dependency, raw-input retention, or public-contract change.
- Initial failures are visible and actionable; expectations are never changed
  merely to reproduce current behavior.
