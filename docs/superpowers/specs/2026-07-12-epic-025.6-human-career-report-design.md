# Epic-025.6 Human Career Report Design

## 1. Purpose and scope

Epic-025.6 adds an internal, user-oriented report layer above NextMove's existing
career intelligence. The report answers four practical questions in a coherent
story: who the user is now, what evidence supports their strengths, how the
target role fits, and what to do next.

This Epic designs presentation and aggregation only. It does not change or
reimplement resume analysis, matching, career-stage assessment, evidence
explanation, transferable-skill assessment, transition assessment, risk
assessment, or transition actions.

The following boundaries are frozen:

- `JobMatchResult`, including its six fields and serialized keys;
- `CareerAnalysisReport` and every existing Skill response;
- `NextMoveSkill.run()` and all public Skill signatures;
- Agent input/output schemas and contracts;
- Application Layer, CLI, providers, frontend, and backend;
- dependency manifests, database behavior, and network behavior.

`HumanCareerReport` is internal to Skill Core. It is not exported from
`skill/__init__.py`, `skill/schemas/`, or `skill/agent/`, and it is not embedded
in an existing public response.

## 2. Current output experience

### 2.1 `CareerAnalysisReport`

There are two frozen orchestration reports. The Skill schema adds execution
state around `analysis`, `improvement`, `job_match`, and `career_advice`; the
Application schema combines `resume_analysis`, `improvement`, `job_match`, and
`career_advice` inside `ApplicationResponse`. Both are useful to machines, but
their fields are parallel capability results rather than a reading journey. A
user must infer how career level, match gaps, and advice relate. Neither report
exposes the richer internal stage, explanation, or transition assessments, and
this Epic changes neither schema.

### 2.2 `JobMatchResult`

`JobMatchResult` is intentionally compact and stable. A score and six lists are
easy to serialize but do not distinguish direct evidence, transferable
evidence, insufficient evidence, and an actual capability conflict. The score
can be over-read as a verdict, while recommendations have no explicit time
horizon or priority.

### 2.3 `CareerTransitionAssessment`

`CareerTransitionAssessment` contains the right internal facts: current and
target domains, transition type, direct and transferable evidence, structured
gaps, risk, actions, and confidence. Its enum values and nested dataclasses are
still an internal decision record. Presenting them directly would make an
ordinary user decode taxonomy, evidence status, risk factors, and numeric
priorities.

### 2.4 User experience gaps

The combined output is machine-oriented, lacks a career story, makes evidence
hard to connect to claims, and does not order actions as a progression. The
Human Report therefore needs:

1. a five-section narrative with fixed reading order;
2. evidence attached to every displayed strength;
3. clear separation of verified gaps from missing information;
4. transition language that distinguishes direct and transferable experience;
5. action horizons ordered from immediate to long term;
6. confidence that means evidence sufficiency, never candidate quality;
7. non-fatalistic language and explicit validation prompts;
8. stable structured serialization for non-Agent internal consumers.

## 3. Design alternatives

### 3.1 Selected: typed internal aggregation model

Add focused internal report schemas and a deterministic builder. This preserves
structure, makes field provenance testable, and provides a stable `to_dict()`
without changing public schemas. The extra internal types are justified because
they prevent presentation strings from becoming an untestable flat blob.

### 3.2 Rejected: one rendered Markdown string

A single template would be quick to display but would make JSON consumers parse
headings and prose. Evidence linkage, action ordering, confidence, and language
safety would be difficult to validate independently.

### 3.3 Rejected: extend `CareerAnalysisReport` or `JobMatchResult`

Adding a report field to either public model would alter serialized Skill and
Agent-visible contracts. It also couples the user report lifecycle to APIs that
this Epic explicitly freezes.

## 4. Internal architecture and data flow

The report layer is a downstream consumer:

```text
ResumeAnalysisResult -----------+
JobMatchResult -----------------+
CareerStageAssessment ----------+--> HumanCareerReportBuilder
MatchExplanationResult ---------+             |
CareerTransitionAssessment -----+             v
                                      HumanCareerReport
                                              |
                                              v
                                           to_dict()
```

The builder receives completed assessments; it never calls a matcher,
classifier, stage assessor, explanation builder, or transition assessor. The
caller remains responsible for composing those existing capabilities. This
keeps the report layer deterministic and prevents it from becoming a second
career-intelligence engine.

Recommended internal package boundaries:

```text
skill/reporting/
    __init__.py      Internal package marker; no root-level export
    schemas.py       Human report enums and immutable dataclasses
    language.py      Fixed zh-CN copy rules and generated-text safety checks
    builder.py       Provenance-aware aggregation and action ordering
    formatter.py     Explicit internal JSON and five-section Markdown output
```

The first implementation targets concise Simplified Chinese copy because the
specified reading structure is Chinese. Field names and enum values remain
stable English identifiers. Locale selection, free-form LLM rewriting, HTML,
and frontend integration are outside this Epic. The internal formatter is an
explicit caller utility and does not alter any existing default output.

## 5. `HumanCareerReport` model

All report models are frozen, slotted dataclasses. Tuples preserve deterministic
ordering and discourage downstream mutation.

```python
class ReportConfidenceLevel(str, Enum):
    HIGH = "high"
    MEDIUM = "medium"
    LOW = "low"


class JobFitLevel(str, Enum):
    STRONG = "strong"
    MODERATE = "moderate"
    EXPLORATORY = "exploratory"
    INSUFFICIENT_EVIDENCE = "insufficient_evidence"


class ActionHorizon(str, Enum):
    IMMEDIATE = "immediate"
    SHORT_TERM = "short_term"
    LONG_TERM = "long_term"


@dataclass(frozen=True, slots=True)
class ReportEvidence:
    text: str
    source: str
    confidence: ReportConfidenceLevel


@dataclass(frozen=True, slots=True)
class CurrentCareerProfile:
    current_domain: str
    core_capabilities: tuple[str, ...]
    profile_summary: str


@dataclass(frozen=True, slots=True)
class CareerStageNarrative:
    stage: str
    explanation: str
    evidence_signals: tuple[str, ...]
    confidence: ReportConfidenceLevel


@dataclass(frozen=True, slots=True)
class ReportStrength:
    capability: str
    explanation: str
    evidence: tuple[ReportEvidence, ...]


@dataclass(frozen=True, slots=True)
class CapabilityGap:
    capability: str
    status: str
    explanation: str
    evidence_needed: str


@dataclass(frozen=True, slots=True)
class JobFitNarrative:
    match_score: int
    level: JobFitLevel
    summary: str
    why_fit: tuple[str, ...]
    main_gaps: tuple[str, ...]


@dataclass(frozen=True, slots=True)
class TransitionPathNarrative:
    current_domain: str
    target_domain: str
    transition_type: str
    summary: str
    transferable_capabilities: tuple[str, ...]
    missing_capabilities: tuple[str, ...]


@dataclass(frozen=True, slots=True)
class ReportAction:
    horizon: ActionHorizon
    priority: int
    related_gap: str
    objective: str
    steps: tuple[str, ...]
    expected_evidence: str


@dataclass(frozen=True, slots=True)
class ActionPlan:
    immediate: tuple[ReportAction, ...]
    short_term: tuple[ReportAction, ...]
    long_term: tuple[ReportAction, ...]


@dataclass(frozen=True, slots=True)
class ReportRisk:
    note: str
    basis: tuple[str, ...]
    verification: str


@dataclass(frozen=True, slots=True)
class ReportConfidence:
    level: ReportConfidenceLevel
    explanation: str
    missing_information: tuple[str, ...]


@dataclass(frozen=True, slots=True)
class HumanCareerReport:
    career_summary: str
    current_profile: CurrentCareerProfile
    career_stage: CareerStageNarrative
    strengths: tuple[ReportStrength, ...]
    job_fit: JobFitNarrative
    capability_gaps: tuple[CapabilityGap, ...]
    transition_path: TransitionPathNarrative
    action_plan: ActionPlan
    risks: tuple[ReportRisk, ...]
    confidence: ReportConfidence

    def to_dict(self) -> dict[str, object]:
        return to_dict(self)
```

The ten top-level fields match the requested report concepts. Nested types keep
evidence, gaps, actions, and confidence machine-consumable instead of hiding
them inside paragraphs.

## 6. Aggregation contract and provenance

```python
HumanCareerReportBuilder.build(
    resume_analysis: ResumeAnalysisResult,
    job_match: JobMatchResult,
    career_stage: CareerStageAssessment,
    match_explanation: MatchExplanationResult,
    career_transition: CareerTransitionAssessment,
) -> HumanCareerReport
```

The inputs are required. Insufficient information is represented by the
existing `unknown`, empty evidence, empty requirements, and low-confidence
values, not by `None` or an exception.

Field provenance is fixed:

| Report field | Authoritative source | Aggregation rule |
|---|---|---|
| `career_summary` | all five inputs | Summarize known domain, stage, fit, and transition; omit unknown claims. |
| `current_profile` | transition current domain, analysis, explanation | Include only capabilities corroborated by explanation evidence; never infer a title. |
| `career_stage` | `CareerStageAssessment` | Explain stage from supplied signals; low confidence when stage is unknown. |
| `strengths` | `MatchExplanationResult.strengths` | Display only items with at least one `EvidenceItem`; preserve direct versus transferable wording. |
| `job_fit` | `JobMatchResult`, explanation | Preserve score; derive reasons from evidence-backed strengths and gaps from explanation. |
| `capability_gaps` | explanation gaps/risks, transition gaps | Deduplicate by canonical capability; keep missing, partial, unknown, and development-area meanings distinct. |
| `transition_path` | `CareerTransitionAssessment` | Translate current-to-target, type, transferability, and missing capabilities without recalculation. |
| `action_plan` | transition recommended actions | Stable sort by numeric priority and original order; map priority 1 to immediate, 2 to short term, and 3+ to long term. Empty horizons remain present. |
| `risks` | explanation risks, transition risk | Describe evidence conditions and verification needs, not suitability or success probability. |
| `confidence` | stage, explanation, transition | Conservative minimum of available evidence channels; list missing inputs that lowered it. |

`JobFitLevel` is a presentation label, not a new score. With at least one target
requirement, scores map deterministically: 80-100 `strong`, 60-79 `moderate`,
and 0-59 `exploratory`. With no extracted requirements, the level is always
`insufficient_evidence`, regardless of score. The score itself is copied
unchanged from `JobMatchResult`.

Confidence never upgrades an existing input. `low` in any required evidence
channel yields report `low`; otherwise any `medium` yields `medium`; only all
high/adequate channels yield `high`. Confidence communicates support for the
report, not employability.

## 7. User reading structure

The structured fields are presented by the internal Markdown formatter in this
fixed order. JSON formatting returns the same structured model without adding
an envelope.

### 第一部分：你的职业画像

Answers “你是谁？” with current domain, evidence-supported core capabilities,
and a plain-language career-stage explanation. Unknown values are stated as
information limitations, never guessed.

### 第二部分：你的优势

Each strength contains a capability, a short explanation, and one or more
evidence records. A `MatchExplanationResult` item with no evidence cannot become
a displayed strength. Transferable evidence must be labelled as transferable
and must not be rewritten as direct target-role experience.

### 第三部分：岗位匹配

Shows the unchanged numeric score, a non-verdict fit level, evidence-backed
reasons, and the main gaps. Unknown requirements are described as “尚未获得证据”
rather than “缺少能力”.

### 第四部分：如果转型

Shows `当前领域 → 目标领域`, the existing transition type, transferable
capabilities, and missing capabilities. For same-role users, this section
reinforces continuity and remaining evidence needs rather than inventing a
transition problem.

### 第五部分：下一步行动

Shows the three containers in order:

```text
立即行动
   ↓
短期提升
   ↓
长期发展
```

Actions retain `related_gap`, ordered steps, and expected evidence from
`TransitionAction`. The report does not create independent career actions.
An empty horizon is serialized as an empty list and rendered as “当前暂无基于证据的建议”.

## 8. Language safety policy

The report uses bounded deterministic templates. Generated narrative is checked
before construction completes; source evidence is copied only when it can be
shown without turning a source claim into a report conclusion.

Forbidden generated conclusions include:

- “你一定成功” and equivalent certainty claims;
- “你不适合” or a score interpreted as personal suitability;
- “你无法转型” and other fatalistic transition claims;
- “保证拿 offer” or any employment-outcome guarantee;
- transferable experience stated as direct target-role experience;
- unknown evidence stated as a proven deficit.

Preferred patterns include:

- “当前证据显示……”;
- “现有经历支持……”;
- “这项能力与目标要求存在可迁移关系，建议验证……”;
- “简历中尚未看到直接证据……”;
- “可以优先补充……”;
- “匹配分数反映当前材料与岗位要求的覆盖情况，不代表录用结果。”

The policy validates generated narrative fields and action copy. It must not
silently rewrite an `EvidenceItem`, because changing evidence would break
grounding. If an evidence snippet itself contains a forbidden guarantee or
fatalistic conclusion, the builder excludes that snippet and does not publish
the corresponding strength unless another safe evidence item remains.

## 9. Scenario design

### Scenario A: strong same-role match

Input: HR Specialist → HR Specialist. Reuse the fictional resume evidence from
`01-hr-specialist.json` with the test-local target “Recruitment Specialist role
requiring recruitment and attendance.” This keeps both classified job families
at `recruitment` and exercises the existing `same_role` path without changing
the shared benchmark fixture.

Expected report emphasis:

- current HR domain and developing stage are understandable;
- recruitment and attendance are shown as strengths with source evidence;
- the fit section emphasizes supported requirements;
- the copied match score is at least 80 and the report label is `strong`;
- transition is `same_role` and does not manufacture a migration narrative;
- an empty action plan is presented honestly rather than filled with invented
  development work.

### Scenario B: cross-domain transition

Input: Sales → Product Manager, using
`03-sales-to-product-manager.json`.

Expected report emphasis:

- the story explicitly shows Sales → Product;
- customer discovery, requirement collection, commercial analysis, and
  stakeholder coordination remain transferable, not direct product ownership;
- Product Planning, PRD, and Product Delivery Ownership remain visible gaps;
- transition risk is explained as sparse direct evidence plus core gaps;
- immediate and short-term actions preserve gap linkage and expected artifacts;
- no claim says the user is unsuitable or unable to transition.

### Scenario C: insufficient information

Input: the empty fictional resume and generic target from
`05-information-insufficient-resume.json`.

Expected report emphasis:

- domain and stage remain unknown;
- no strengths, missing skills, or transition feasibility are invented;
- job fit is `insufficient_evidence` even if a default score exists;
- report confidence is low and names the missing resume and target-role detail;
- next steps ask for responsibilities, skills, outcomes, dates, and specific
  target requirements before stronger conclusions are drawn.

## 10. Serialization and compatibility

`HumanCareerReport.to_dict()` delegates to the existing recursive
`skill.utils.to_dict` utility. Enums serialize to their string values, tuples to
JSON arrays, and nested dataclasses to JSON objects. The resulting dictionary
must pass `json.dumps()` without custom encoders.

`HumanCareerReportFormatter.to_json()` serializes that dictionary with Unicode
preserved. `to_markdown()` renders the five fixed reading sections plus nested
risk and confidence details. Both are internal explicit calls; neither is wired
into Agent, Application, CLI, backend, or frontend behavior.

Serialization is additive and internal. Existing calls to
`to_dict(CareerAnalysisReport(...))`, Agent tools, `NextMoveSkill.run()`, the
Application Layer, and CLI produce byte-for-byte compatible key sets and value
shapes. No automatic report generation is added to those paths in this Epic.

## 11. Validation and regression strategy

Implementation validation will cover:

1. exact `HumanCareerReport` top-level field order and JSON serialization;
2. provenance tests proving every displayed strength has evidence;
3. status-preserving gap mapping and direct/transferable separation;
4. deterministic job-fit labels, confidence aggregation, and action ordering;
5. language-policy tests for forbidden and preferred phrasing;
6. the three user scenarios above, including weak-input non-fabrication;
7. public contract tests for `JobMatchResult`, Agent schemas,
   `CareerAnalysisReport`, `NextMoveSkill.run()`, Application Layer, and CLI;
8. import-boundary tests ensuring `skill/reporting/` is not imported by public
   schemas or Agent modules.

The full existing test suite, Career Intelligence benchmark, compilation, JSON
serialization, and `git diff --check` remain required before implementation
closeout.

## 12. Non-goals

- exposing the report through Agent, Application, CLI, backend, or frontend;
- adding locale negotiation, HTML rendering, or frontend presentation;
- invoking an LLM to rewrite the report;
- changing match scores, career stages, transition types, risks, or actions;
- storing reports or sending them over a network;
- updating Epic closeout files during this design-only commit.
